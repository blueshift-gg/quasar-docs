---
title: Associated Token Accounts
description: Deterministic token accounts derived from wallet + mint.
---

ATAs are deterministic token accounts derived from a wallet address and a mint (seeds: `[wallet, token_program, mint]`, program: ATA program). Quasar provides declarative initialization, address validation, and derivation helpers.

## Declarative Initialization

Use `init` with the `associated_token(...)` behavior to create an ATA. Initialization names the token, System, and ATA program fields used by the CPI:

```rust
use quasar_spl::{
    accounts::associated_token,
    AssociatedTokenProgram, Mint, Token, TokenCpi, TokenProgram,
};

#[derive(Accounts)]
pub struct InitAta {
    #[account(mut)]
    pub payer: Signer,
    #[account(
        init,
        associated_token(
            mint = mint,
            authority = wallet,
            token_program = token_program,
            system_program = system_program,
            ata_program = ata_program
        )
    )]
    pub ata: Account<Token>,
    pub wallet: Signer,
    pub mint: Account<Mint>,
    pub token_program: Program<TokenProgram>,
    pub system_program: Program<SystemProgram>,
    pub ata_program: Program<AssociatedTokenProgram>,
}
```

Use `init(idempotent)` to create the account if it doesn't exist. For an existing account, Quasar validates that the mint and authority match:

```rust
#[account(
    init(idempotent),
    associated_token(
        mint = mint,
        authority = wallet,
        token_program = token_program,
        system_program = system_program,
        ata_program = ata_program
    )
)]
pub ata: Account<Token>,
```

### `associated_token(...)` vs `token(...)`

`associated_token(...)` creates an ATA at the deterministic PDA address through the ATA program. `token(...)` creates a generic token account through `InitializeAccount3`. Use `associated_token(...)` when you need the canonical ATA address.

## Validation Without Init

To validate an existing ATA without creating it, use `associated_token(...)` without `init`. Validation needs the mint and authority; pass `token_program` to bind an interface account to a specific token program. The System and ATA programs are creation-only arguments.

```rust
#[account(associated_token(
    mint = mint,
    authority = wallet,
    token_program = token_program
))]
pub ata: Account<Token>,
```

This derives the expected address, checks that the account matches, and validates the token data.

## Address Derivation

For computing ATA addresses in instruction logic:

```rust
use quasar_spl::get_associated_token_address;

let (ata_address, bump) = get_associated_token_address(&wallet, &mint);
```

For a specific token program (e.g., Token-2022):

```rust
use quasar_spl::get_associated_token_address_with_program;

let (ata_address, bump) = get_associated_token_address_with_program(&wallet, &mint, &token_program_id);
```

For `const` contexts:

```rust
use quasar_spl::get_associated_token_address_const;

const ATA: (Address, u8) = get_associated_token_address_const(&WALLET, &MINT);
```

## Closing ATAs

ATAs are regular token accounts. Close them through the token program, passing the account, lamport destination, and authority in that order:

```rust
self.token_program
    .close_account(&self.ata, &self.destination, &self.authority)
    .invoke()?;
```
