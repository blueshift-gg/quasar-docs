---
title: Quasar
description: A zero-copy Rust framework for Solana programs, with account validation generated at compile time.
---

Quasar is a Rust framework for Solana programs built around direct account
views. Its macros validate account metadata and layout before a handler runs;
the handler then works on the SVM input buffer without deserializing account
data into owned structs.

That model removes copies and heap work from the instruction path, but it also
makes layout and validation part of the program's safety boundary. Quasar keeps
those contracts explicit through `#[derive(Accounts)]`, `#[account]`, typed PDA
seeds, and generated IDLs and clients.

<Callout type="warn" title="Beta software">
Quasar v0.1.0 is unaudited beta software. Do not use it to hold or control real
funds. Start with local programs and review the [zero-copy safety
model](/docs/zero-copy/safety-model) before relying on direct account views.
</Callout>

## Understand the model

- **Accounts are borrowed views.** Quasar checks the input region, alignment,
  ownership, signer state, and declared constraints before exposing typed
  account data. See [Accounts and validation](/docs/core-concepts/accounts-and-validation).
- **Layouts are contracts.** Account discriminators, versions, padding, and
  dynamic fields determine what can be read safely and upgraded later. See
  [Account layout](/docs/zero-copy/account-layout).
- **Generated artifacts share one schema.** The IDL drives typed Rust and
  TypeScript clients, so client encoding follows the same instruction and
  account definitions as the program. See [IDL](/docs/core-concepts/idl).
- **Tests run without a validator.** QuasarSVM executes program binaries in
  process from Rust or TypeScript. See [Testing](/docs/testing/overview).
