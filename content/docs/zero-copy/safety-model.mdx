Quasar uses `unsafe` for zero-copy account access, raw CPI syscalls, pointer arithmetic over the SVM input buffer, and `MaybeUninit` patterns for stack-allocated arrays. This page catalogs the unsafe patterns, explains the soundness argument for each, and describes the compile-time and runtime protections.

## Unsafe inventory

The `quasar-lang` crate contains `unsafe` blocks and trait impls across five pattern categories:

1. **Zero-copy pointer casts** (`&[u8]` to `&ZcType`)
2. **Interior mutability** (`&` to `&mut` casts)
3. **Raw CPI syscalls** (`sol_invoke_signed_c`)
4. **Pointer arithmetic** (remaining accounts, entrypoint dispatch)
5. **`MaybeUninit` patterns** (stack-allocated arrays)

The following sections explain each category and the invariants that make it sound.

## Zero-copy pointer casts

The core operation in Quasar is casting a raw byte buffer to a `#[repr(C)]` companion struct. Every `Deref` on an account view type performs this cast:

```rust
impl core::ops::Deref for Escrow {
    type Target = EscrowZc;

    fn deref(&self) -> &Self::Target {
        unsafe {
            &*(self.__view.data_ptr().add(DISC_LEN) as *const EscrowZc)
        }
    }
}
```

This pattern also appears in `Sysvar<T>`, `Account<T>`, and the `ZeroCopyDeref` trait.

**Soundness** relies on three invariants:

1. **`#[repr(transparent)]`**: Account view types (`Account<T>`, `Sysvar<T>`) are `#[repr(transparent)]` wrappers over `AccountView`. The cast from `&AccountView` to `&Account<T>` is a no-op at the memory level.

2. **Alignment 1**: ZC companion structs use Pod types (`PodU64`, `PodU128`, etc.) which are `#[repr(transparent)]` over `[u8; N]`. Compile-time assertions enforce `align_of::<ZcStruct>() == 1`. Since the source data (`&[u8]`) also has alignment 1, the pointer cast is always aligned.

3. **Bounds checked upstream**: `AccountCheck::check()` validates `data_len >= size_of::<ZcStruct>() + discriminator_len` before any pointer cast occurs. The `Deref` impl performs no bounds validation itself -- it relies entirely on this upstream check.

## Interior mutability

The critical aliasing pattern: `from_account_view_unchecked_mut` produces an `&mut Account<T>` that can coexist with `&AccountView` references pointing to the same `RuntimeAccount`:

```rust
// Generated by the #[account] macro and in Account<T>
unsafe fn from_account_view_unchecked_mut(view: &mut AccountView) -> &mut Self {
    &mut *(view as *mut AccountView as *mut Self)
}
```

While the function signature takes `&mut AccountView`, in practice the caller may hold other `&AccountView` references derived from the same `RuntimeAccount` pointer. This means the `&mut Account<T>` aliases memory accessible through shared references. The soundness argument:

1. `Account<T>` is `#[repr(transparent)]` over `T`, which is `#[repr(transparent)]` over `AccountView`, which holds a raw `*mut RuntimeAccount` pointer.
2. All mutations (setting lamports, writing account data, calling `assign`) go through `AccountView`'s raw pointer methods, not through Rust's mutable reference semantics.
3. The `&mut Account<T>` reference exists solely to provide `DerefMut` access to the ZC companion struct, which writes through the same raw pointers.
4. Under the **Tree Borrows** model, the `&mut` retag creates a "Reserved" permission. Since no write occurs through the mutable reference to `AccountView`'s own fields, the permission never transitions to "Active", and no undefined behavior occurs.

Every call site checks `is_writable()` before performing the cast, returning `ProgramError::Immutable` for read-only accounts.

### Tree Borrows dependency

This pattern is UB under the **Stacked Borrows** model but sound under **Tree Borrows**. Quasar explicitly depends on Tree Borrows semantics:

- **Miri must be run with `-Zmiri-tree-borrows`**. Under the default Stacked Borrows model, the `& -> &mut` casts are flagged as UB.
- The Rust compiler does not currently commit to either aliasing model. Tree Borrows is the likely successor to Stacked Borrows, but it is not yet stabilized.
- This is an accepted risk, shared with **Pinocchio** and other Solana frameworks. The entire ecosystem relies on this pattern for zero-copy mutable account access.

A future major version could eliminate this dependency by wrapping `AccountView`'s inner pointer in `UnsafeCell`, making the pattern formally sound under any model at zero runtime cost.

## Raw CPI syscalls

Quasar builds CPI calls by constructing `RawCpiBuilder` structs on the stack, which are layout-compatible with `CpiAccount` and transmuted before passing to `sol_invoke_signed_c`. The CPI path is guarded by compile-time assertions:

```rust
const _: () = assert!(core::mem::size_of::<RawCpiBuilder>() == 56);
const _: () = assert!(core::mem::size_of::<RawCpiBuilder>() == core::mem::size_of::<CpiAccount>());
const _: () = assert!(core::mem::align_of::<RawCpiBuilder>() == core::mem::align_of::<CpiAccount>());
```

The 56-byte size assertion catches any field addition or reordering at compile time. The flag extraction (reading `is_signer`, `is_writable`, `executable` from `RuntimeAccount`) relies on `offset_of!` assertions:

```rust
const _: () = assert!(core::mem::offset_of!(RuntimeAccount, borrow_state) == 0);
const _: () = assert!(core::mem::offset_of!(RuntimeAccount, is_signer) == 1);
const _: () = assert!(core::mem::offset_of!(RuntimeAccount, is_writable) == 2);
const _: () = assert!(core::mem::offset_of!(RuntimeAccount, executable) == 3);
```

These protect against upstream layout changes in the `solana-account-view` crate.

## Pointer arithmetic

The remaining accounts system and entrypoint dispatch walk the SVM input buffer using raw pointer arithmetic. The SVM serialization format is:

```
[account_count: u64]
[RuntimeAccount header | data | padding to 8-byte alignment] * N
[ix_data_len: u64]
[ix_data bytes]
[program_id: 32 bytes]
```

Every pointer dereference in the remaining accounts iterator is preceded by a **boundary check**: `ptr as *const u8 >= self.boundary`. The boundary pointer marks the end of the accounts region (the `ix_data_len` field), computed once at entrypoint dispatch.

The alignment round-up `((ptr as usize + 7) & !7)` involves an integer round-trip that loses pointer provenance under `-Zmiri-strict-provenance`. This is not UB under the default provenance model and matches the standard Solana deserialization pattern used by all frameworks.

## MaybeUninit patterns

Six locations use `MaybeUninit` for stack-allocated arrays:

| Location | Buffer | Init pattern |
|----------|--------|-------------|
| CPI accounts | `[CpiAccount; N]` | `while i < N` loop writes exactly `N` elements |
| System instructions | `[u8; 52]`, `[u8; 12]`, `[u8; 36]` | `create_account` (52), `transfer` (12), `assign` (36) written via `copy_nonoverlapping` |
| Iterator cache | `[AccountView; 64]` | Elements read via `ptr::read` at validated positions only |
| Declared accounts | `[AccountView; COUNT]` | `parse_accounts` writes exactly `COUNT` elements |
| Sysvar buffer | `MaybeUninit<Self>` | Syscall fills the buffer; padding zeroed explicitly |
| PDA derivation | `MaybeUninit<[u8; 32]>`, `MaybeUninit<[&[u8]; 19]>` | `assume_init` only on syscall success |

The iterator cache is notable: `assume_init` is never called on the full array. Instead, individual elements are read via `ptr::read` at positions `0..self.index` -- elements that were previously written by prior `next()` calls. A bounds check prevents reading uninitialized memory:

```rust
if remaining_idx >= self.index {
    return None;
}
```

## Compile-time assertions catalog

Quasar uses `const _: () = assert!(...)` extensively to catch invariant violations at compile time:

| Assertion | Purpose |
|-----------|---------|
| `align_of::<ZcStruct>() == 1` | Every ZC companion struct has alignment 1 |
| `size_of::<RawCpiBuilder>() == 56` | CPI builder struct matches `CpiAccount` layout |
| `size_of::<Rent>() == 16`, `align_of::<Rent>() == 1` | Sysvar struct size and alignment |
| `size_of::<Clock>() == 40`, `align_of::<Clock>() == 1` | Sysvar struct size and alignment |
| Discriminator non-zero | All-zero discriminators banned (prevents uninitialized data matching) |
| No padding in event struct | `size_of::<Event>() == computed_data_size` ensures memcpy serialization is safe |
| Instruction args alignment 1 | ZC arg structs safe for pointer cast from `&[u8]` |
| Return type alignment 1 | Return data pointer cast is aligned |
| Discriminator != `0xFF` | Reserved for self-CPI event prefix |
| No discriminator collisions | Same-kind collisions are compile errors |

These assertions form a defense-in-depth layer that catches bugs before any unsafe code runs.

## Miri validation

Every unsafe code path is tested under Miri with Tree Borrows and symbolic alignment checking:

```bash
MIRIFLAGS="-Zmiri-tree-borrows -Zmiri-symbolic-alignment-check" \
  cargo +nightly miri test -p quasar-lang --test miri
```

The test suite covers 79 tests across all unsafe categories:

- Aliasing casts through `from_account_view_unchecked_mut`
- `copy_nonoverlapping` flag extraction in CPI account construction
- `MaybeUninit` array initialization in `CpiCall::new`
- Event `memcpy` serialization (no padding, alignment 1)
- CPI data construction (system program `create_account`, `transfer`)
- Remaining accounts pointer arithmetic (non-duplicate and duplicate resolution)
- Dynamic field probes: exact-size buffer boundaries, max-capacity fields, shared-to-mut aliasing under Tree Borrows retag, minimum-overlap memmove geometry, and instruction data ZC cast at exact Vec boundaries

## Attack mitigations

### All-zero discriminator attack

An attacker creates an account with all-zero data (the default for new accounts). If a program used all-zero discriminators, uninitialized accounts would pass validation.

**Mitigation**: All-zero discriminators are banned at compile time. The `#[account]` macro generates an assertion that at least one discriminator byte is non-zero. Since uninitialized account data is all-zero, it can never match a valid discriminator.

### Re-initialization attack

An attacker calls an init instruction on an account that already contains valid data, overwriting it.

**Mitigation**: The generated `init()` method checks that the discriminator region is all-zero bytes before writing. Combined with the all-zero discriminator ban, this creates a clean invariant: valid accounts always have non-zero discriminators, and `init()` only succeeds on accounts with zero discriminators (uninitialized).

### Account revival attack

After closing an account within a transaction, a subsequent instruction re-opens it. If the data was not zeroed during close, the old discriminator persists.

**Mitigation**: `Account::close()` zeros the discriminator bytes **before** draining lamports:

```rust
pub fn close(&mut self, destination: &AccountView) -> Result<(), ProgramError> {
    let view = unsafe { &mut *(self as *mut Account<T> as *mut AccountView) };
    if !destination.is_writable() {
        return Err(ProgramError::Immutable);
    }

    // Zero discriminator bytes to prevent revival
    let zero_len = view.data_len().min(8);
    unsafe { core::ptr::write_bytes(view.data_mut_ptr(), 0, zero_len) };

    let new_lamports = destination.lamports().wrapping_add(view.lamports());
    set_lamports(destination, new_lamports);
    view.set_lamports(0);

    unsafe { view.assign(&SYSTEM_PROGRAM_ID) };
    resize(view, 0)?;
    Ok(())
}
```

The destination is checked for writability, the discriminator is zeroed first, then lamports drained (using `wrapping_add` since total SOL supply fits in `u64`), then owner reassigned to the system program and data resized to zero. Even within the same transaction, the account cannot pass discriminator validation after close.

### Misaligned access

Account data from the SVM has alignment 1. Casting to a struct with alignment > 1 is undefined behavior.

**Mitigation**: All zero-copy types use Pod fields with alignment 1. Every ZC companion struct has a compile-time assertion: `assert!(align_of::<ZcStruct>() == 1)`. This applies to account state structs, instruction argument structs, and sysvar structs. The `RawCpiBuilder` struct has alignment 8, but it is constructed on the stack -- never cast from byte data.

### Duplicate account chain overflow

Malformed SVM input with chained duplicate references could cause excessive iteration.

**Mitigation**: `resolve_dup_walk` uses iterative resolution with a **2-hop depth limit**. The SVM guarantees duplicates resolve in one hop (a dup always points to a non-dup). The depth limit provides defense-in-depth without risking stack overflow on the 4KB SBF stack.

## CPI without borrow checks

Quasar's CPI path does **not** check or modify `RuntimeAccount::borrow_state` before invoking `sol_invoke_signed_c`. This is deliberate, not an oversight.

Three invariants make borrow checks unnecessary:

1. **Raw pointer access only**: Quasar accesses data through `data_ptr()` / `data_mut_ptr()` and raw pointer casts -- never through `try_borrow()` / `try_borrow_mut()`. The `borrow_state` field is never modified from its initial `NOT_BORROWED` value.

2. **No-duplicate enforcement**: The `parse_accounts` codegen checks byte 0 of the `RuntimeAccount` header is `0xFF` (`NOT_BORROWED`). Duplicates have a small index number in byte 0, so the check fails. This prevents two declared `AccountView`s from aliasing the same `RuntimeAccount`.

3. **Zero-copy eliminates sync gap**: The traditional SDK copies data between `RuntimeAccount` and local buffers, requiring serialize-before-CPI and deserialize-after-CPI. Quasar writes directly to `RuntimeAccount` -- changes are always in-place. The callee sees the latest state, and after CPI returns, the caller sees the callee's changes.

## Zero heap allocation

The `no_alloc!()` macro installs a global allocator that panics on any heap allocation:

```rust
pub struct NoAlloc;
unsafe impl GlobalAlloc for NoAlloc {
    unsafe fn alloc(&self, _: Layout) -> *mut u8 {
        panic!("");
    }
    unsafe fn dealloc(&self, _: *mut u8, _: Layout) {}
}
```

This is a runtime guarantee that the entire dispatch-parse-CPI path is zero-allocation. Any accidental `Vec`, `String`, or `Box` usage triggers a panic during testing. The `heap_alloc!()` macro provides the standard Solana bump allocator for programs that need heap access.
