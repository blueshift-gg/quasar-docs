---
title: Remaining Accounts
description: Parsing bounded, typed account lists after an instruction's declared accounts.
---

Some instructions need a variable number of accounts, such as a multisig signer set or a list of accounts forwarded to another program. Remaining accounts are the entries passed after the declared fields in a `#[derive(Accounts)]` struct.

## `Ctx` vs `CtxWithRemaining`

- `Ctx<T>` parses only the instruction's declared accounts.
- `CtxWithRemaining<T>` also preserves the trailing account region.

Choose the context in the instruction signature, then parse the trailing entries into a bounded type before calling business logic:

```rust
#[instruction(discriminator = 0)]
pub fn create(ctx: CtxWithRemaining<Create>, threshold: u8) -> Result<(), ProgramError> {
    let signers = ctx.remaining_accounts().parse::<Signer, 10>()?;
    ctx.accounts.create_multisig(threshold, &ctx.bumps, signers)
}
```

The `#[program]` macro detects `CtxWithRemaining<T>` and keeps the trailing account bytes available through `ctx.remaining_accounts()`.

## Typed parsing with `Remaining<T, N>`

`parse::<T, N>()` validates every remaining item and returns `Remaining<T, N>`, a zero-allocation collection stored in a fixed-capacity stack array. `N` is the maximum number of typed items, not a runtime allocation size.

```rust
let signers: Remaining<Signer, 10> =
    ctx.remaining_accounts().parse::<Signer, 10>()?;

if signers.is_empty() {
    return Err(ProgramError::NotEnoughAccountKeys);
}

for signer in signers.iter() {
    let address: &Address = signer.address();
    // use address...
}
```

The item type applies its normal account validation during parsing. `Signer` checks the transaction signer flag; `Account<T>` checks owner and account data; `Program<T>`, `InterfaceAccount<T>`, sysvars, and custom `#[derive(Accounts)]` groups use their corresponding validators.

Parsing fails before business logic receives the value when an entry does not satisfy `T`, when a fixed-count group is incomplete, when the input contains more than `N` typed items, or when parsing would consume more than 64 raw accounts. A custom accounts group can consume multiple raw accounts per typed item, but still occupies one slot in `Remaining<T, N>`.

| Method | Result |
|--------|--------|
| `as_slice()` | All parsed items as `&[T]` |
| `iter()` | Iterator over `&T` |
| `len()` | Number of parsed typed items |
| `capacity()` | Compile-time capacity `N` |
| `is_empty()` | Whether no items were parsed |

## Raw access

Use the raw `RemainingAccounts` accessor only when the tail is heterogeneous or requires manual validation. It provides:

```rust
let remaining = ctx.remaining_accounts();

if let Some(account) = remaining.get(0)? {
    require!(account.is_writable(), ProgramError::InvalidAccountData);
}

for account in remaining.iter() {
    let account = account?;
    // Apply the exact checks required by this heterogeneous entry.
}
```

`get(index)` walks from the beginning of the remaining region. The iterator advances once, caches previously yielded views for duplicate resolution, and returns `RemainingAccountsOverflow` after 64 raw entries.

## Multisig creation

The instruction boundary parses at most ten signers:

```rust
#[instruction(discriminator = 0)]
pub fn create(ctx: CtxWithRemaining<Create>, threshold: u8) -> Result<(), ProgramError> {
    let signers = ctx.remaining_accounts().parse::<Signer, 10>()?;
    ctx.accounts.create_multisig(threshold, &ctx.bumps, signers)
}
```

The accounts handler receives only validated `Signer` values:

```rust
#[derive(Accounts)]
pub struct Create {
    #[account(mut)]
    pub creator: Signer,
    #[account(init, payer = creator, address = MultisigConfig::seeds(creator.address()))]
    pub config: Account<MultisigConfig>,
    pub rent: Sysvar<Rent>,
    pub system_program: Program<SystemProgram>,
}

impl Create {
    pub fn create_multisig(
        &mut self,
        threshold: u8,
        bumps: &CreateBumps,
        signers: Remaining<Signer, 10>,
    ) -> Result<(), ProgramError> {
        let mut addrs = core::mem::MaybeUninit::<[Address; 10]>::uninit();
        let addrs_ptr = addrs.as_mut_ptr() as *mut Address;
        let mut count = 0usize;

        for signer in signers.iter() {
            // SAFETY: typed parsing guarantees count is below the capacity 10.
            unsafe { core::ptr::write(addrs_ptr.add(count), *signer.address()) };
            count = count.wrapping_add(1);
        }

        let threshold_count = threshold as usize;
        if threshold_count == 0 || threshold_count > count {
            return Err(ProgramError::InvalidArgument);
        }

        // SAFETY: Elements 0..count were initialized by the loop above.
        let signer_addresses = unsafe { core::slice::from_raw_parts(addrs_ptr, count) };

        self.config.set_inner(
            MultisigConfigInner {
                creator: *self.creator.address(),
                threshold,
                bump: bumps.config,
                label: "",
                signers: signer_addresses,
            },
            self.creator.to_account_view(),
            self.rent.lamports_per_byte(),
            self.rent.exemption_threshold_raw(),
        )
    }
}
```

The parser owns the signer check and ten-item bound. The handler only copies validated addresses into the compact account field and enforces the multisig threshold rule.

## Multisig execution

The execute path uses the same typed boundary:

```rust
#[instruction(discriminator = 3)]
pub fn execute_transfer(
    ctx: CtxWithRemaining<ExecuteTransfer>,
    amount: u64,
) -> Result<(), ProgramError> {
    let signers = ctx.remaining_accounts().parse::<Signer, 10>()?;
    ctx.accounts
        .verify_and_transfer(amount, &ctx.bumps, signers)
}
```

```rust
impl ExecuteTransfer {
    pub fn verify_and_transfer(
        &self,
        amount: u64,
        bumps: &ExecuteTransferBumps,
        signers: Remaining<Signer, 10>,
    ) -> Result<(), ProgramError> {
        let stored_signers = self.config.signers();
        let threshold = self.config.threshold as u32;

        let mut approvals = 0u32;
        for stored in stored_signers {
            for signer in signers.iter() {
                if quasar_lang::keys_eq(signer.address(), stored) {
                    approvals = approvals.wrapping_add(1);
                    break;
                }
            }
            if approvals >= threshold {
                break;
            }
        }

        if approvals < threshold {
            return Err(ProgramError::MissingRequiredSignature);
        }

        let vault_signer = self.vault_signer(bumps);
        self.system_program
            .transfer(&self.vault, &self.recipient, amount)
            .invoke_signed(&vault_signer)?;
        Ok(())
    }
}
```

`Signer` permits repeated account metas during typed parsing. Iterating stored signers first and breaking after the first match ensures a repeated transaction meta cannot count as multiple approvals for one configured signer.

## Duplicate handling and cost

Raw iteration resolves duplicate SVM entries against declared accounts and previously yielded remaining accounts. Typed parsing applies the duplicate policy of `T`; remaining item types reject duplicate addresses by default, while `Signer` allows them for workflows that handle repetition explicitly.

| Operation | Cost |
|-----------|------|
| `ctx.remaining_accounts()` | O(1) pointer construction |
| `parse::<T, N>()` | O(n) parsing, plus O(n²) duplicate checks when `T` rejects duplicates |
| `Remaining<T, N>::iter()` | O(1) per typed item |
| Raw `remaining.get(n)` | O(n) walk from the start |
| Raw `remaining.iter().next()` | O(1) amortized |

Prefer typed parsing for homogeneous account lists. It centralizes validation, makes the maximum item count explicit in the type, and gives handlers infallible iteration over validated values. Keep raw access for genuinely heterogeneous tails and document every manual check.
