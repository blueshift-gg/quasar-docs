title: CLI Reference

---

The `quasar` CLI builds, tests, profiles, and deploys Solana programs. It wraps the Solana/Upstream toolchain and manages IDL generation and sBPF disassembly.

## Install

```bash
cargo install quasar-cli --version 0.1.0 --locked
```

---

## Commands

### `quasar init`

Scaffold a new Quasar project.

```
quasar init [NAME] [OPTIONS]
```

Without `--yes`, the command is interactive. A positional name becomes the default value for the project-name prompt. With `--yes`, a name is required and the CLI uses explicit flags followed by saved or built-in defaults for unspecified choices.

| Flag | Description |
|------|-------------|
| `NAME` | Project name (becomes crate name and `Quasar.toml` project name). Use `.` to scaffold into the current directory. |
| `-y, --yes` | Skip all prompts and use explicit flags, then saved or built-in defaults |
| `--no-git` | Skip `git init` |
| `--test-language <LANGUAGE>` | Test language: `none`, `rust`, or `typescript` |
| `--rust-framework <FRAMEWORK>` | Rust framework when the test language is `rust`: `quasar-svm` or `mollusk` |
| `--ts-sdk <SDK>` | Solana SDK when the test language is `typescript`: `kit` or `web3.js` |
| `--template <TEMPLATE>` | Project template: `minimal` (single instruction) or `full` (state, events, instruction files) |
| `--toolchain <TOOLCHAIN>` | Build toolchain: `solana` or `upstream` |
| `--verbose` | Show each scaffold step as it runs |

Generates a project with `Cargo.toml`, `Quasar.toml`, source files, test scaffolding, program keypair, and git setup. Preferences are saved to `~/.quasar/config.toml`.

**Examples:**

```bash
quasar init                        # Interactive wizard
quasar init my-program             # Default the name; prompt for remaining choices
quasar init my-program --yes       # Use saved or built-in defaults without prompts
quasar init . --toolchain upstream # Scaffold into current directory
quasar init my-app --verbose       # Show each generated step
quasar init my-app --test-language rust --rust-framework mollusk --template full
quasar init my-app --test-language typescript --ts-sdk kit --template full
```

### `quasar add`

Generate instructions, state accounts, or error enums in the current project.

```
quasar add [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `-i, --instruction <NAME>` | Create `src/instructions/<name>.rs`, wire its module, and add an auto-discriminated entry to the `#[program]` module |
| `-s, --state <NAME>` | Create or append an `#[account]` struct in `src/state.rs` using the next account discriminator |
| `-e, --error <NAME>` | Create or append an `#[error_code]` enum in `src/errors.rs` |

At least one flag is required, and the flags can be combined. Names may contain
hyphens; Quasar converts them to underscores before checking that they are valid
Rust identifiers.

**Examples:**

```bash
quasar add --instruction withdraw
quasar add --state vault
quasar add --error access-error
quasar add -i deposit -s vault -e vault-error
```

Creates `src/instructions/withdraw.rs`:

```rust
use quasar_lang::prelude::*;

#[derive(Accounts)]
pub struct Withdraw {
    pub payer: Signer,
    pub system_program: Program<SystemProgram>,
}

impl Withdraw {
    #[inline(always)]
    pub fn withdraw(&self) -> Result<(), ProgramError> {
        Ok(())
    }
}
```

And adds to `src/lib.rs`:

```rust
#[instruction]
pub fn withdraw(ctx: Ctx<Withdraw>) -> Result<(), ProgramError> {
    ctx.accounts.withdraw()
}
```

### `quasar build`

Compile the program. Reads `Quasar.toml` for the toolchain and generates the IDL before building.

```
quasar build [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `--debug` | Emit debug symbols (required for profiling and source-interleaved dump) |
| `--verbose` | Stream the underlying build command output |
| `-w, --watch` | Watch `src/` for changes and rebuild automatically |
| `--features <FEATURES>` | Cargo features to enable (passed through to the build command) |

On success, prints the binary size and delta from the previous build:

```
  Build complete in 1.2s (56.6 KB, -1.2 KB)
```

**Examples:**

```bash
quasar build                  # Release build
quasar build --debug          # Debug build with symbols
quasar build --verbose        # Stream compiler output
quasar build --watch          # Auto-rebuild on changes
quasar build --features "devnet"
```

### `quasar test`

Run the test suite. Builds first (unless `--no-build`), then runs Rust or TypeScript tests based on `Quasar.toml`.

```
quasar test [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `--debug` | Build with debug symbols before testing |
| `--show-output` | For Cargo tests, forward `--show-output` to the test harness |
| `-f, --filter <PATTERN>` | Only run tests whose name matches the pattern |
| `-w, --watch` | Watch `src/` for changes and re-run tests automatically |
| `--no-build` | Skip the build step (use existing binary) |
| `--features <FEATURES>` | Cargo features to enable during the build step |
| `--verbose` | Show the build and test commands as they run |

Test framework behavior:

- **Rust** (`testing.language = "rust"`): Runs the command in `[testing.rust.test]`.
- **TypeScript** (`testing.language = "typescript"`): Runs `[testing.typescript.install]` when `node_modules` is absent, then runs `[testing.typescript.test]`.

**Examples:**

```bash
quasar test                       # Build and test
quasar test --filter deposit      # Only run tests matching "deposit"
quasar test --watch               # Auto-test on changes
quasar test --no-build            # Skip build, test existing binary
quasar test --debug --filter pda  # Debug build, filtered tests
quasar test --features "devnet"    # Build with Cargo features, then test
quasar test --show-output         # Show output captured by Cargo tests
quasar test --verbose             # Print the commands being run
```

### `quasar profile`

Estimate static compute cost from the sBPF call graph and measure the deploy
binary size. Quasar runs a debug build when you omit the ELF path.

```
quasar profile [ELF] [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `ELF` | Path to an unstripped `.so`; omit it to build and detect the program |
| `--expand` | Print the complete function table |
| `--diff <PROGRAM>` | Compare stored profile snapshots for a program |
| `--share` | Upload the profile as a public GitHub Gist |
| `-w, --watch` | Rebuild when `src/` changes |
| `--budget <FILE>` | Budget file for write or assertion mode; defaults to `quasar-budget.toml` |
| `--write-budget` | Write ceilings for the current program |
| `--assert-budget` | Exit with code 2 when a ceiling is exceeded |
| `--headroom <PERCENT>` | Add headroom when writing ceilings; defaults to 5 |
| `--json` | Print stable machine-readable output without starting the flamegraph server |

`--write-budget` records the static total, selected function costs, and the
stripped deploy ELF size. Commit the file, then enforce it in CI:

```bash
quasar profile --write-budget --headroom 5
quasar profile --assert-budget --json
```

The static CU estimate counts sBPF instructions along the resolved call graph.
It is useful for detecting code-generation regressions. Runtime transaction CU
also includes syscall costs, account work, and the path taken by a specific
instruction, so measure that separately in program tests.

Without `--json`, Quasar stores profile snapshots under `target/profile/`
and starts the local flamegraph viewer. See [CU Profiler](/docs/profiling/cu-profiler)
for the budget format and interpretation.

**Examples:**

```bash
quasar profile
quasar profile --expand
quasar profile --write-budget --headroom 10
quasar profile --assert-budget --budget ci/quasar-budget.toml
quasar profile --json > profile.json
quasar profile target/deploy/debug/my_program.so.debug
```

### `quasar dump`

Dump sBPF assembly using `llvm-objdump`. Auto-detects the ELF from `target/deploy/` or `target/profile/` if not specified.

```
quasar dump [ELF] [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `ELF` | Path to a compiled `.so` (auto-detected if omitted) |
| `-f, --function <SYMBOL>` | Disassemble only the named symbol (demangled name) |
| `-S, --source` | Interleave source code (requires a debug build) |

Prints an instruction count summary at the end.

**Examples:**

```bash
quasar dump                              # Full disassembly
quasar dump --function initialize        # Single function
quasar dump --function initialize -S     # With source interleaving
quasar dump target/deploy/my_program.so  # Specific ELF
```

### `quasar deploy`

Build, preflight, deploy, and verify the program. Quasar uses the cluster and
payer from your Solana CLI config unless a flag overrides them.

```
quasar deploy [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `--program-keypair <KEYPAIR>` | Program keypair; defaults to `target/deploy/<name>-keypair.json` |
| `--upgrade-authority <KEYPAIR>` | Upgrade authority keypair; defaults to the Solana CLI signer |
| `-k, --keypair <KEYPAIR>` | Transaction payer; defaults to the Solana CLI signer |
| `-u, --url <URL>` | Cluster moniker or RPC URL |
| `--skip-build` | Deploy the existing binary after running the same lint and lock checks |
| `--skip-verify` | Skip the post-deploy byte and authority check |

Programs with generated instruction discriminators need a checked-in
`quasar.lock.json`. Create or update it with `quasar lint --update-lock`.
A normal deployment then builds the program, deploys it through the Solana CLI,
dumps the cluster bytes, and compares them with the local ELF.

Successful verification writes
`target/deploy/<program>-deployment.json`. The manifest records program,
artifact, ABI, authority, source, and cluster provenance. Quasar resolves a
missing `--url` from the Solana CLI configuration and marks a dirty source
revision with a `-dirty` suffix.

**Examples:**

```bash
quasar deploy
quasar deploy --skip-build
quasar deploy --url devnet --keypair ~/.config/solana/id.json
quasar deploy --program-keypair ./my-key.json
quasar deploy --upgrade-authority ./authority.json
```

See [Deploy and verify a program](/docs/guides/deploy-and-verify) for the release
workflow and the limits of the verification.

### `quasar verify`

Compare local artifacts and expected deployment provenance with a live program.

```
quasar verify [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `--program-id <ADDRESS>` | Program address; defaults to the project keypair address |
| `--program-keypair <KEYPAIR>` | Program keypair used to derive the address |
| `--upgrade-authority <KEYPAIR>` | Keypair whose address must match the live upgrade authority |
| `-u, --url <URL>` | Cluster moniker or RPC URL |
| `--elf-path <ELF>` | Local ELF; defaults to `target/deploy/<name>.so` |
| `--manifest <MANIFEST>` | Deployment manifest; Quasar detects the default path when omitted |

The command checks the program ID, deployed ELF hash, IDL hash, ABI hash,
upgrade authority, program-data address, last deploy slot, cluster, and source
revision represented by the manifest. It reports every manifest mismatch in one
run.

```bash
quasar verify --url devnet
quasar verify --program-id <ADDRESS> --elf-path artifacts/program.so --url devnet
quasar verify --manifest artifacts/program-deployment.json --url devnet
```

### `quasar clean`

Remove generated deploy, profile, IDL, and client artifacts. Files ending in
`-keypair.json` are preserved in `target/deploy/` because replacing one changes
the program address.

```
quasar clean [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `-a, --all` | Also run `cargo clean` to remove the complete Cargo build output |

```bash
quasar clean       # Remove Quasar-generated artifacts
quasar clean --all # Also remove Cargo build output
```

### `quasar idl`

Generate an IDL and Rust client from a program crate, or verify the hashes stored
in an existing Quasar IDL.

```
quasar idl <PATH>
quasar idl verify <IDL>
```

| Argument or subcommand | Description |
|------------------------|-------------|
| `PATH` | Path to the program crate directory |
| `verify <IDL>` | Recompute `hashes.idl` and `hashes.abi` and compare them with the values stored in the IDL file |

Verification detects content or ABI drift after the hashes were created. It
does not prove who produced the file or that the program is deployed on-chain.

**Examples:**

```bash
quasar idl .                            # Generate from the current crate
quasar idl programs/my-program          # Generate from a specific crate
quasar idl verify target/idl/program.json # Check stored IDL and ABI hashes
```

### `quasar client`

Generate language clients from an existing IDL file.

```
quasar client <IDL> [--lang <LANGUAGES>]
```

| Argument or flag | Description |
|------------------|-------------|
| `IDL` | Path to a generated IDL JSON file |
| `--lang <LANGUAGES>` | Comma-separated subset of `typescript`, `python`, or `golang`; omit it to generate all three |

**Examples:**

```bash
quasar client target/idl/my_program.json --lang typescript
quasar client target/idl/my_program.json --lang python,golang
```

### `quasar lint`

Audit the current program surface for pre-deploy and upgrade-safety findings.
When `quasar.lock.json` exists, lint also compares the generated IDL with the
recorded surface.

```
quasar lint [OPTIONS]
```

| Flag | Description |
|------|-------------|
| `--update-lock` | Write the current program surface to `quasar.lock.json` after a successful lint |
| `--no-diff` | Run current-surface rules without comparing against `quasar.lock.json` |
| `--strict` | Treat warning and informational findings as failures |

The lockfile records an API surface for later comparison; it is not a program
binary, deployment record, or security proof.

```bash
quasar lint               # Check current rules and any recorded surface
quasar lint --update-lock # Record the reviewed surface
quasar lint --no-diff     # Ignore the existing surface for this run
quasar lint --strict      # Fail on every reported severity
```

### `quasar audit`

Print the account validation and lifecycle plan generated by the compiler.

```
quasar audit [IDL] [--json]
```

| Argument or flag | Description |
|------------------|-------------|
| `IDL` | Generated IDL JSON; omit it to use the current project's IDL |
| `--json` | Print the versioned validation-plan extension as JSON |

For each instruction, the report shows account access, wrapper and load mode,
rent handling, checks before and after load, and epilogue work such as `close`.
Use it during review to confirm what the macro generated from an accounts
struct.

```bash
quasar build
quasar audit
quasar audit target/idl/my_program.json --json > validation-plan.json
```

The report describes generated mechanics. It does not decide whether your
authority model or business constraints are sufficient.

### `quasar keys`

Inspect or manage the program keypair under `target/deploy/`.

```
quasar keys <SUBCOMMAND>
```

| Subcommand | Description |
|------------|-------------|
| `list` | Print the program ID derived from the current keypair file |
| `sync` | Update `declare_id!()` in `src/lib.rs` to match the current keypair |
| `new [--force]` | Generate a new keypair and sync `declare_id!()`; refuse to replace an existing keypair unless `--force` is present |

`quasar keys new --force` changes the program address. Use it only when replacing
the current program identity is intentional.

```bash
quasar keys list
quasar keys sync
quasar keys new
quasar keys new --force
```

### `quasar config`

Manage global settings in `~/.quasar/config.toml`. Running without a subcommand opens an interactive menu.

```
quasar config [SUBCOMMAND]
```

| Subcommand | Description |
|------------|-------------|
| `get <KEY>` | Read a single config value |
| `set <KEY> <VALUE>` | Write a config value |
| `list` | Print every config value |
| `reset` | Restore factory defaults |

**Available keys:**

| Key | Valid Values | Default |
|-----|-------------|---------|
| `defaults.toolchain` | `solana`, `upstream` | (not set) |
| `defaults.test_language` | `none`, `rust`, `typescript` | (not set) |
| `defaults.rust_framework` | `quasar-svm`, `mollusk` | (not set) |
| `defaults.ts_sdk` | `kit`, `web3.js` | (not set) |
| `defaults.template` | `minimal`, `full` | (not set) |
| `ui.animation` | `true`, `false` | `true` |
| `ui.color` | `true`, `false` | `true` |

**Examples:**

```bash
quasar config                        # Interactive menu
quasar config list                   # Print all settings
quasar config get ui.animation       # Read a value
quasar config set ui.color false     # Write a value
quasar config set defaults.toolchain solana
quasar config reset                  # Restore factory defaults
```
