title: Cross-Program Invocations
description: Stack-allocated CPI with const-generic account and data sizes — zero heap allocation by design.
---

Cross-program invocations (CPI) in Quasar are built around two const-generic types that keep everything on the stack. No heap allocation, no intermediate borrow checking — just raw pointer arithmetic straight to the `sol_invoke_signed_c` syscall.

## CpiCall: Fixed-Size CPI

`CpiCall<'a, const ACCTS: usize, const DATA: usize>` is the primary CPI type. Both the account count and instruction data size are known at compile time:

```rust
pub struct CpiCall<'a, const ACCTS: usize, const DATA: usize> {
    program_id: &'a Address,
    accounts: [InstructionAccount<'a>; ACCTS],
    cpi_accounts: [CpiAccount<'a>; ACCTS],
    data: [u8; DATA],
}
```

Everything lives in stack-allocated fixed-size arrays. The compiler knows the exact memory layout at build time, which means zero runtime overhead for buffer management.

### Invocation Methods

Every `CpiCall` provides three invocation methods:

```rust
// No PDA signers needed
cpi_call.invoke()?;

// Single PDA signer (most common)
cpi_call.invoke_signed(&seeds)?;

// Multiple PDA signers
cpi_call.invoke_with_signers(&[signer_a, signer_b])?;
```

Under the hood, all three call `invoke_raw`, which passes raw pointers directly to `sol_invoke_signed_c` — the SVM syscall for CPI. There is no intermediate `Instruction` struct, no `Vec` allocation, no borrow checking overhead.

## BufCpiCall: Variable-Length CPI

When instruction data has a variable length (e.g., Borsh-serialized payloads), use `BufCpiCall<'a, const ACCTS: usize, const MAX: usize>`:

```rust
pub struct BufCpiCall<'a, const ACCTS: usize, const MAX: usize> {
    program_id: &'a Address,
    accounts: [InstructionAccount<'a>; ACCTS],
    cpi_accounts: [CpiAccount<'a>; ACCTS],
    data: [u8; MAX],
    data_len: usize,
}
```

The buffer has a compile-time maximum capacity (`MAX`) but tracks the actual data length at runtime. The constructor validates `data_len <= MAX` and panics if exceeded. This gives you variable-length semantics without heap allocation.

`BufCpiCall` exposes the same `invoke()`, `invoke_signed()`, and `invoke_with_signers()` methods as `CpiCall`.

## System Program Helpers

Quasar provides typed helper functions for the most common System program CPIs. These return `CpiCall` instances with exact sizes:

| Function | Type | Description |
|----------|------|-------------|
| `transfer` | `CpiCall<2, 12>` | Transfer lamports between accounts |
| `create_account` | `CpiCall<2, 52>` | Create a new account with space and owner |
| `assign` | `CpiCall<1, 36>` | Assign an account to a new owner program |
| `create_account_with_minimum_balance` | `CpiCall<2, 52>` | Create with rent-exempt balance (fetches Rent sysvar) |

### Using Free Functions

The free functions accept raw `&AccountView` references:

```rust
use quasar_lang::cpi::system;

system::transfer(from_view, to_view, 1_000_000).invoke()?;
system::create_account(payer_view, new_view, lamports, space, &owner).invoke()?;
system::assign(account_view, &new_owner).invoke()?;
```

### Using Program Methods

More commonly, you call methods on `Program<System>`, which accepts any type implementing `AsAccountView`:

```rust
// From the vault example: simple SOL transfer
self.system_program
    .transfer(self.user, self.vault, amount)
    .invoke()?;

// Create account with minimum rent-exempt balance
self.system_program
    .create_account_with_minimum_balance(
        self.payer,
        self.new_account,
        space,
        &owner,
        Some(&self.rent),
    )?
    .invoke()?;
```

The `Program<System>` methods convert each account to an `AccountView` via `to_account_view()` before constructing the `CpiCall`. This means you can pass `&Account<T>`, `&Signer`, `&UncheckedAccount`, or any other Quasar account type directly.

## PDA Signing with Bumps

When a CPI requires a PDA signer, use the `Bumps` struct generated by `#[derive(Accounts)]`. It provides `*_seeds()` methods that return properly formatted seed slices.

Here is a real example from the escrow program's `take` instruction, where the escrow PDA must sign for token transfers:

```rust
impl<'info> Take<'info> {
    pub fn withdraw_tokens_and_close(&mut self, bumps: &TakeBumps) -> Result<(), ProgramError> {
        let seeds = bumps.escrow_seeds();

        // Transfer tokens from vault to taker, signed by the escrow PDA
        self.token_program
            .transfer(
                self.vault_ta_a,
                self.taker_ta_a,
                self.escrow,
                self.vault_ta_a.amount(),
            )
            .invoke_signed(&seeds)?;

        // Close the vault token account, signed by the escrow PDA
        self.vault_ta_a
            .close(self.token_program, self.taker, self.escrow)
            .invoke_signed(&seeds)
    }
}
```

The `TakeBumps` struct is auto-generated from the `Take` accounts struct (see [Accounts and Validation](./accounts-and-validation)). The `escrow_seeds()` method returns the seeds array `[b"escrow", maker_bytes, &[bump]]` ready for the CPI signer.

## Token Program CPI

The `quasar-spl` crate wraps the SPL Token program with typed CPI helpers. Methods are available on `Program<Token>`:

```rust
// Transfer tokens (no PDA signer)
self.token_program
    .transfer(self.taker_ta_b, self.maker_ta_b, self.taker, amount)
    .invoke()?;

// Transfer tokens (PDA signer)
self.token_program
    .transfer(self.vault_ta_a, self.taker_ta_a, self.escrow, amount)
    .invoke_signed(&seeds)?;
```

The token account's `close` method is also CPI-based:

```rust
self.vault_ta_a
    .close(self.token_program, self.destination, self.authority)
    .invoke_signed(&seeds)?;
```

All SPL CPI helpers follow the same `CpiCall` pattern — stack-allocated, const-generic, invoked through `.invoke()` or `.invoke_signed()`.

## declare_program! — Typed CPI from IDL

The `declare_program!` macro generates a typed CPI module from a program's IDL JSON at compile time. This enables type-safe cross-program calls without manually constructing instruction data.

```rust
declare_program!(my_program, "path/to/program.idl.json");
```

This generates:

- A module named `my_program`
- A program type (e.g., `MyProgram`) with `Id` and address checks
- A `const ID: Address` for the program's address
- Free functions for each instruction that return `CpiCall` instances
- Methods on the program type that accept `&impl AsAccountView`

### Generated CPI Functions

For an IDL instruction like:

```json
{
  "name": "Transfer",
  "discriminator": [2],
  "accounts": [
    { "name": "from", "writable": true, "signer": true },
    { "name": "to", "writable": true }
  ],
  "args": [
    { "name": "amount", "type": "u64" }
  ]
}
```

The macro generates:

```rust
pub mod my_program {
    pub const ID: Address = address!("...");

    // Free function variant
    pub fn transfer<'a>(
        __program: &'a AccountView,
        from: &'a AccountView,
        to: &'a AccountView,
        amount: u64,
    ) -> CpiCall<'a, 2, 12> {
        // Writes discriminator + args into a fixed-size buffer
        // Constructs InstructionAccount entries with correct flags
        // Returns a ready-to-invoke CpiCall
    }

    // Method variant on the program type
    impl MyProgram {
        pub fn transfer<'a>(
            &'a self,
            from: &'a impl AsAccountView,
            to: &'a impl AsAccountView,
            amount: u64,
        ) -> CpiCall<'a, 2, 12> { /* ... */ }
    }
}
```

The data buffer size is computed at macro expansion time from the discriminator length plus the sum of all argument byte sizes. Account flags (`writable`, `signer`) are set from the IDL metadata.

### Supported Argument Types

`declare_program!` supports fixed-size primitive types:

| IDL Type | Rust Type | Size |
|----------|-----------|------|
| `u8`, `i8`, `bool` | `u8`, `i8`, `bool` | 1 byte |
| `u16`, `i16` | `u16`, `i16` | 2 bytes |
| `u32`, `i32` | `u32`, `i32` | 4 bytes |
| `u64`, `i64` | `u64`, `i64` | 8 bytes |
| `u128`, `i128` | `u128`, `i128` | 16 bytes |
| `pubkey` | `&Address` | 32 bytes |

Defined types (structs) are not supported in CPI arguments — only fixed-size primitives and addresses.

## Events via CPI

Quasar supports two event emission mechanisms:

### Log-Based Events (~100 CU)

The `emit!` macro serializes the event and writes it to the program log:

```rust
emit!(MakeEvent {
    escrow: *self.escrow.address(),
    maker: *self.maker.address(),
    mint_a: *self.mint_a.address(),
    mint_b: *self.mint_b.address(),
    deposit,
    receive,
});
```

This is fast but can be spoofed by other programs writing to the same log.

### Self-CPI Events (~1,000 CU)

For tamper-proof events, Quasar uses a self-CPI pattern. The `#[program]` macro generates an `EventAuthority` PDA (seeds: `["__event_authority"]`) computed at compile time. The event discriminator `0xFF` prefix is reserved for this protocol.

When a self-CPI event is emitted, the program invokes itself with the event data, signed by the `EventAuthority` PDA. Because the program ID appears in the transaction trace, clients can verify the event was genuinely emitted by the program.

## How CPI Works Internally

Under the hood, `CpiCall::invoke()` calls `invoke_raw`, which constructs a `#[repr(C)]` instruction struct and passes it directly to the `sol_invoke_signed_c` syscall:

1. **Account conversion**: `cpi_account_from_view` reads the 4-byte header `[borrow_state, is_signer, is_writable, executable]` as a `u32`, shifts right 8 bits to drop `borrow_state`, and stores the three flag bytes. This is a single unaligned read rather than three separate field accesses.

2. **Instruction layout**: The `CInstruction` struct is `#[repr(C)]` and layout-compatible with the C struct expected by the SVM syscall.

3. **No intermediate types**: There is no `solana_instruction::Instruction`, no `Vec<AccountMeta>`, no `to_vec()` on the data. Raw pointers go straight to the syscall.

This design means CPI overhead is essentially just the syscall cost itself.
