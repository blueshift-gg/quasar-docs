---
title: Cross-Program Invocations
description: Stack-allocated CPI with const-generic account and data sizes — zero heap allocation by design.
---

Cross-program invocations (CPI) in Quasar use two const-generic types that keep everything on the stack: `CpiCall` for fixed-size instruction data, and `BufCpiCall` for variable-length data. Both go directly to the `sol_invoke_signed_c` syscall with no intermediate `Instruction` struct or `Vec` allocation.

## CpiCall: Fixed-Size CPI

```rust
pub struct CpiCall<'a, const ACCTS: usize, const DATA: usize> {
    program_id: &'a Address,
    accounts: [InstructionAccount<'a>; ACCTS],
    cpi_accounts: [CpiAccount<'a>; ACCTS],
    data: [u8; DATA],
}
```

Both the account count and instruction data size are const generics, so the entire struct is stack-allocated with a known layout at compile time.

Three invocation methods:

```rust
cpi_call.invoke()?;                            // No PDA signers
cpi_call.invoke_signed(&seeds)?;               // Single PDA signer
cpi_call.invoke_with_signers(&[s1, s2])?;      // Multiple PDA signers
```

All three call `invoke_raw`, which constructs a `#[repr(C)]` `CInstruction` and passes raw pointers directly to `sol_invoke_signed_c`.

## BufCpiCall: Variable-Length CPI

For variable-length instruction data (e.g., Borsh-serialized payloads), use `BufCpiCall`:

```rust
pub struct BufCpiCall<'a, const ACCTS: usize, const MAX: usize> {
    program_id: &'a Address,
    accounts: [InstructionAccount<'a>; ACCTS],
    cpi_accounts: [CpiAccount<'a>; ACCTS],
    data: [u8; MAX],
    data_len: usize,
}
```

The buffer has compile-time maximum capacity `MAX` but tracks actual data length at runtime. The constructor panics if `data_len > MAX`. Same invocation methods as `CpiCall`.

## System Program Helpers

Free functions in `quasar_lang::cpi::system` and methods on `Program<System>`:

| Function | Return Type | Description |
|----------|-------------|-------------|
| `transfer` | `CpiCall<2, 12>` | Transfer lamports (discriminator `2`) |
| `create_account` | `CpiCall<2, 52>` | Create account with space and owner (discriminator `0`) |
| `assign` | `CpiCall<1, 36>` | Reassign account to a new owner (discriminator `1`) |
| `create_account_with_minimum_balance` | `Result<CpiCall<2, 52>, ProgramError>` | Create with rent-exempt balance (fetches Rent sysvar if not provided) |

Free functions accept `&AccountView` references:

```rust
use quasar_lang::cpi::system;

system::transfer(from_view, to_view, 1_000_000).invoke()?;
system::create_account(payer_view, new_view, lamports, space, &owner).invoke()?;
system::assign(account_view, &new_owner).invoke()?;
```

`Program<System>` methods accept `&impl AsAccountView`, so you can pass `&Account<T>`, `&Signer`, `&UncheckedAccount`, etc. directly:

```rust
self.system_program
    .transfer(self.user, self.vault, amount)
    .invoke()?;

self.system_program
    .create_account_with_minimum_balance(
        self.payer,
        self.new_account,
        space,
        &owner,
        Some(&*self.rent),  // Option<&Rent>
    )?
    .invoke()?;
```

## PDA Signing with Bumps

When a CPI requires a PDA signer, use the `Bumps` struct generated by `#[derive(Accounts)]`. Its `*_seeds()` methods return `[Seed; N]` arrays ready for `invoke_signed`.

From the escrow program's `take` instruction:

```rust
impl<'info> Take<'info> {
    pub fn withdraw_tokens_and_close(&mut self, bumps: &TakeBumps) -> Result<(), ProgramError> {
        let seeds = bumps.escrow_seeds();

        // Transfer tokens from vault to taker, signed by the escrow PDA
        self.token_program
            .transfer(
                self.vault_ta_a,
                self.taker_ta_a,
                self.escrow,
                self.vault_ta_a.amount(),
            )
            .invoke_signed(&seeds)?;

        // Close the vault token account, signed by the escrow PDA
        self.vault_ta_a
            .close(self.token_program, self.taker, self.escrow)
            .invoke_signed(&seeds)
    }
}
```

The `TakeBumps` struct is auto-generated from the `Take` accounts struct (see [Accounts and Validation](./accounts-and-validation)). The `escrow_seeds()` method returns the seeds array `[b"escrow", maker_bytes, &[bump]]` ready for the CPI signer.

## Token Program CPI

The `quasar-spl` crate provides typed CPI helpers. Methods on `Program<Token>`:

```rust
// Transfer tokens (no PDA signer)
self.token_program
    .transfer(self.taker_ta_b, self.maker_ta_b, self.taker, amount)
    .invoke()?;

// Transfer tokens (PDA signer)
self.token_program
    .transfer(self.vault_ta_a, self.taker_ta_a, self.escrow, amount)
    .invoke_signed(&seeds)?;
```

The token account's `close` method is also CPI-based:

```rust
self.vault_ta_a
    .close(self.token_program, self.destination, self.authority)
    .invoke_signed(&seeds)?;
```

## declare_program! — Typed CPI from IDL

`declare_program!` reads a program's IDL JSON at compile time and generates a typed CPI module:

```rust
declare_program!(my_program, "path/to/program.idl.json");
```

This generates a `pub mod my_program` containing:
- `pub const ID: Address` -- the program address from the IDL
- A program type (e.g., `MyProgram`) with `Executable` + `Address` checks and `Id` impl
- A free function per instruction that accepts `&AccountView` references and returns a `CpiCall`
- Methods on the program type that accept `&impl AsAccountView`

### Generated CPI Functions

For an IDL instruction `"Transfer"` with discriminator `[2]`, two accounts (`from`: writable+signer, `to`: writable), and one arg (`amount: u64`):

```rust
pub mod my_program {
    pub const ID: Address = address!("...");

    // Free function: accounts as &AccountView
    pub fn transfer<'a>(
        __program: &'a AccountView,
        from: &'a AccountView,
        to: &'a AccountView,
        amount: u64,
    ) -> CpiCall<'a, 2, 9> { /* ... */ }

    // Method: accounts as &impl AsAccountView
    impl MyProgram {
        pub fn transfer<'a>(
            &'a self,
            from: &'a impl AsAccountView,
            to: &'a impl AsAccountView,
            amount: u64,
        ) -> CpiCall<'a, 2, 9> { /* ... */ }
    }
}
```

The data buffer size (`9` = 1-byte discriminator + 8-byte u64) is computed at macro expansion time from the discriminator length plus the sum of all argument byte sizes. Account flags (`writable`, `signer`) are set from IDL metadata.

### Supported Argument Types

`declare_program!` supports fixed-size primitive types:

| IDL Type | Rust Type | Size |
|----------|-----------|------|
| `u8`, `i8`, `bool` | `u8`, `i8`, `bool` | 1 byte |
| `u16`, `i16` | `u16`, `i16` | 2 bytes |
| `u32`, `i32` | `u32`, `i32` | 4 bytes |
| `u64`, `i64` | `u64`, `i64` | 8 bytes |
| `u128`, `i128` | `u128`, `i128` | 16 bytes |
| `pubkey` | `&Address` | 32 bytes |

Defined types (structs) are not supported in CPI arguments — only fixed-size primitives and addresses.

## Events

Two emission mechanisms:

### Log-Based: `emit!` (~100 CU)

Calls `emit_log()` on the event, which serializes it and writes it to the program log via `sol_log_data`:

```rust
emit!(MakeEvent {
    escrow: *self.escrow.address(),
    maker: *self.maker.address(),
    mint_a: *self.mint_a.address(),
    mint_b: *self.mint_b.address(),
    deposit,
    receive,
});
```

Fast but spoofable -- other programs can write to the same log.

### Self-CPI: `emit_cpi!` (~1,000 CU)

The `emit_cpi!` macro expands to `self.program.emit_event(&event, self.event_authority, crate::EventAuthority::BUMP)`. This requires the accounts struct to have `program` and `event_authority` fields. The program invokes itself with the event data prefixed by `0xFF`, signed by the `EventAuthority` PDA (seeds: `["__event_authority"]`). The `__handle_event` dispatch handler verifies the event authority's signature and address, then logs the event data.

Because the program ID appears in the transaction trace, clients can verify the event was genuinely emitted by the program.

## Internals

`CpiCall::invoke()` calls `invoke_raw`, which:

1. **Account flag extraction**: `cpi_account_from_view` reads the 4-byte header `[borrow_state, is_signer, is_writable, executable]` as a `u32`, shifts right 8 bits to drop `borrow_state`, and stores the three flag bytes in a `RawCpiBuilder` struct. This is then transmuted to `CpiAccount` (verified layout-compatible by compile-time assertions).

2. **Syscall dispatch**: A `CInstruction` struct (`#[repr(C)]`, layout-compatible with the C struct expected by `sol_invoke_signed_c`) is constructed on the stack and passed directly to the syscall. No `solana_instruction::Instruction`, no `Vec<AccountMeta>`, no data serialization.
