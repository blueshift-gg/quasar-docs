title: IDL Generation
description: How Quasar generates program interface descriptions and client code from your Rust source.

The IDL is a JSON file describing your program's interface -- instructions, accounts, events, types, and errors. Generated from your Rust source by the CLI, it drives TypeScript and Rust client generation, and is consumed by [`declare_program!`](./cpi#declare_program) for typed CPI at compile time.

## What the IDL Contains

```json
{
  "spec": "quasar-idl/1.0.0",
  "name": "quasar_escrow",
  "version": "0.1.0",
  "address": "22222222222222222222222222222222222222222222",
  "metadata": {
    "crateName": "quasar-escrow",
    "generatorVersion": "0.1.0",
    "schemaVersion": "1.0.0"
  },
  "instructions": [],
  "accounts": [],
  "events": [],
  "types": [],
  "errors": []
}
```

`spec` identifies the IDL format and schema family. The top-level `version` is the program crate version. `metadata` records the Cargo package name, the Quasar generator version, and the schema version separately so consumers do not have to infer one version from another.

### Instructions

Each instruction includes its name, discriminator bytes, accounts with flags, and typed arguments:

```json
{
  "name": "make",
  "discriminator": [0],
  "accounts": [
    {
      "name": "maker",
      "writable": true,
      "signer": true,
      "resolver": { "kind": "input" }
    },
    {
      "name": "escrow",
      "writable": true,
      "signer": false,
      "resolver": {
        "kind": "pda",
        "program": { "kind": "programId" },
        "seeds": [
          { "kind": "const", "value": [101, 115, 99, 114, 111, 119] },
          { "kind": "account", "path": "maker" }
        ]
      }
    },
    {
      "name": "mintA",
      "writable": false,
      "signer": false,
      "resolver": { "kind": "input" }
    },
    {
      "name": "makerTaA",
      "writable": true,
      "signer": false,
      "resolver": { "kind": "input" }
    },
    {
      "name": "vaultTaA",
      "writable": true,
      "signer": true,
      "resolver": { "kind": "input" }
    },
    {
      "name": "tokenProgram",
      "writable": false,
      "signer": false,
      "resolver": {
        "kind": "const",
        "address": "TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA"
      }
    },
    {
      "name": "systemProgram",
      "writable": false,
      "signer": false,
      "resolver": {
        "kind": "const",
        "address": "11111111111111111111111111111111"
      }
    }
  ],
  "args": [
    { "name": "deposit", "type": "u64" },
    { "name": "receive", "type": "u64" }
  ],
  "layout": {
    "kind": "fixed",
    "fields": ["deposit", "receive"]
  }
}
```

- Every account node records explicit `writable` and `signer` flags plus a `resolver`.
- Caller-provided accounts use `resolver: { "kind": "input" }`.
- PDA resolvers name the derivation program and encode seeds as `const` bytes or an `account` path.
- Known program addresses use a `const` resolver with an `address`.
- A fixed `layout` lists instruction arguments in wire order.

Instructions using `CtxWithRemaining` include a structured `remainingAccounts` contract:

```json
{
  "remainingAccounts": {
    "kind": "append",
    "name": "remainingAccounts",
    "min": 0,
    "max": null,
    "item": {
      "clientType": "accountMeta",
      "signer": "input",
      "writable": "input"
    },
    "policy": {
      "position": "afterDeclaredAccounts",
      "order": "preserveInput"
    }
  }
}
```

`kind: "append"` and `position: "afterDeclaredAccounts"` place these entries after the declared account list. `order: "preserveInput"` requires generated clients to retain caller order. The caller supplies each account meta and its signer and writable flags. `min: 0` permits an empty tail, while `max: null` means the IDL itself does not set a maximum. On-chain code can still impose a typed bound with `Remaining<T, N>`.

The normalized v0.1 ABI baseline stores the same object under `remaining_accounts`; generated IDL uses the camel-cased `remainingAccounts` key shown above.

Accounts and events each list their name and discriminator. Event discriminators are prefixed with `0xFF` on the wire to distinguish them from instructions.

### Types

Struct types referenced by instructions or accounts. The only non-obvious mapping is `pubkey` for `Address` -- everything else (`u8`, `u64`, `bool`, etc.) maps directly to its Rust equivalent:

```json
{
  "name": "Escrow",
  "type": {
    "kind": "struct",
    "fields": [
      { "name": "maker", "type": "pubkey" },
      { "name": "mintA", "type": "pubkey" },
      { "name": "receive", "type": "u64" },
      { "name": "bump", "type": "u8" }
    ]
  }
}
```

Dynamic fields have dedicated representations:

```json
{
  "name": "label",
  "type": "string",
  "codec": {
    "kind": "sizePrefixed",
    "prefix": { "type": "u8", "endian": "le" },
    "storage": "tail",
    "maxBytes": 32,
    "encoding": "utf8"
  }
}
{
  "name": "signers",
  "type": { "vec": "pubkey" },
  "codec": {
    "kind": "sizePrefixed",
    "prefix": { "type": "u16", "endian": "le" },
    "storage": "tail",
    "maxItems": 10
  }
}
```

| Rust | IDL type | Default compact codec |
|------|----------|-----------------------|
| `String<MAX>` | `string` | `u8` byte-length prefix, UTF-8 tail, `maxBytes = MAX` |
| `Vec<T, MAX>` | `{ "vec": T }` | `u16` element-count prefix, data tail, `maxItems = MAX` |

Bounded borrowed instruction arguments such as `#[max(32)] label: &str` project to the same compact string or vector IDL shape. Borrowed values are not account field types.

### Errors

`#[error_code]` variants are offset by 6000. Framework errors (`QuasarError`) start at 3000:

```json
{ "code": 6000, "name": "InvalidAmount", "msg": "Invalid amount" }
```

## Generating the IDL

```bash
quasar idl path/to/program
```

### Output structure

```
target/
  idl/
    my_program.json
  client/
    rust/
      my-program-client/
        Cargo.toml
        src/lib.rs                   # Typed instruction builders
```

### TypeScript clients

`quasar build` generates the languages listed in `[clients].languages`. To generate a client explicitly from an existing IDL, run:

```bash
quasar client target/idl/my_program.json --lang typescript
```

The TypeScript output contains two flavors from the same IDL: one for `@solana/web3.js` v2 and one for `@solana/kit`. Both include typed instruction builders, account/event type definitions, discriminator constants, and codec functions.

### Rust client

A standalone crate with typed `Instruction` builders. Each instruction struct implements `From<T> for Instruction`:

```rust
let ix: Instruction = DepositInstruction {
    user: user_pubkey,
    vault: vault_pda,
    system_program: system_program::ID,
    amount: 1_000_000,
}.into();
```
