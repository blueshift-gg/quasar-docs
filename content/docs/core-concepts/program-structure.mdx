The `#[program]` macro generates the on-chain entrypoint, instruction dispatch table, event authority PDA, panic handler, and allocator configuration.

## Module Layout

A typical Quasar program has this directory structure:

```
my-program/
  Cargo.toml
  src/
    lib.rs              # Program entry, declare_id!, #[program] module
    instructions/       # One file per instruction handler
      mod.rs
      deposit.rs
      withdraw.rs
    state.rs            # #[account] definitions
    errors.rs           # #[error_code] enum
    events.rs           # #[event] definitions
```

The escrow example's `lib.rs`:

```rust
#![no_std]

use quasar_lang::prelude::*;

mod instructions;
use instructions::*;
mod events;
mod state;

declare_id!("22222222222222222222222222222222222222222222");

#[program]
mod quasar_escrow {
    use super::*;

    #[instruction(discriminator = 0)]
    pub fn make(ctx: Ctx<Make>, deposit: u64, receive: u64) -> Result<(), ProgramError> {
        ctx.accounts.make_escrow(receive, &ctx.bumps)?;
        ctx.accounts.emit_event(deposit, receive)?;
        ctx.accounts.deposit_tokens(deposit)
    }

    #[instruction(discriminator = 1)]
    pub fn take(ctx: Ctx<Take>) -> Result<(), ProgramError> {
        ctx.accounts.transfer_tokens()?;
        ctx.accounts.withdraw_tokens_and_close(&ctx.bumps)?;
        ctx.accounts.emit_event()
    }

    #[instruction(discriminator = 2)]
    pub fn refund(ctx: Ctx<Refund>) -> Result<(), ProgramError> {
        ctx.accounts.withdraw_tokens_and_close(&ctx.bumps)?;
        ctx.accounts.emit_event()
    }
}
```

- **`#![no_std]`** is required. Quasar programs target the SVM runtime and do not link `std`.
- **`declare_id!`** sets the `crate::ID` constant used for owner checks and PDA derivation.
- **`#[program]`** annotates a module containing `#[instruction]` functions. The module name (`quasar_escrow`) determines the generated program type name (`QuasarEscrow`). A `client` submodule with typed instruction builders is also generated inside the program module (compiled only for non-SVM targets).

## What `#[program]` Generates

### 1. Entrypoint

The macro generates an `entrypoint` function with the `#[unsafe(no_mangle)]` attribute, which the Solana runtime calls when your program is invoked:

```rust
#[unsafe(no_mangle)]
#[cfg(target_os = "solana")]
pub unsafe extern "C" fn entrypoint(ptr: *mut u8, instruction_data: *const u8) -> u64 {
    // Initialize bump allocator cursor (if alloc feature is enabled)
    // Reconstruct instruction_data slice: length is at offset -8 from the pointer
    // Dispatch to __dispatch(ptr, instruction_data)
}
```

`ptr` points to the raw SVM input buffer containing all accounts. `instruction_data` points to the serialized instruction payload, with its `u64` length stored at offset `-8`.

### 2. Dispatch Table

`__dispatch` checks the first byte of instruction data. If it is `0xFF`, the call is routed to `__handle_event` (the self-CPI event handler) instead of the instruction dispatch table. Otherwise, the discriminator prefix is matched against a compile-time table of instruction handlers via the `dispatch!` macro.

Constraints enforced at compile time:
- All instruction discriminators must have the same byte length.
- No duplicates.
- No discriminator may start with `0xFF` (reserved for events).

### 3. Program Type and Event Authority

Outside the module, the macro generates:

```rust
// #[repr(transparent)] wrapper over AccountView, validates Executable + Address
pub struct QuasarEscrow { /* view: AccountView */ }

impl Id for QuasarEscrow {
    const ID: Address = crate::ID;
}

// Event authority PDA, computed at compile time (zero runtime cost)
pub struct EventAuthority { /* view: AccountView */ }

impl EventAuthority {
    const __PDA: (Address, u8) = find_program_address_const(&[b"__event_authority"], &crate::ID);
    pub const ADDRESS: Address = Self::__PDA.0;
    pub const BUMP: u8 = Self::__PDA.1;
}
```

The program type gets an `emit_event` method that takes an `&impl Event` and an `&EventAuthority`, invoking self-CPI with the event data signed by the `EventAuthority` PDA.

### 4. Panic Handler

The macro generates a minimal panic handler (`#[panic_handler]`) that calls `log("PANIC")` and enters an infinite loop. This is gated behind `#[cfg(any(target_os = "solana", target_arch = "bpf"))]`.

## Allocator Configuration

By default, the `#[program]` macro emits `no_alloc!()`, which installs a global allocator that panics on any allocation attempt. Quasar's zero-copy design means most programs never need heap allocation.

If your program requires heap allocation (e.g., for `alloc::vec::Vec` or `alloc::string::String`), enable the `alloc` feature flag:

```toml
[dependencies]
quasar-lang = { version = "...", features = ["alloc"] }
```

With `alloc` enabled, the macro emits `heap_alloc!()` instead, installing a bump allocator. The entrypoint initializes the heap cursor before any allocation can occur:

```rust
#[cfg(feature = "alloc")]
{
    let heap_start = allocator::HEAP_START_ADDRESS as usize;
    *(heap_start as *mut usize) = heap_start + core::mem::size_of::<usize>();
}
```

Memory is never reclaimed within a single instruction execution.

## Debug Mode

Quasar programs can be built with the `debug` feature flag to enable detailed error logging:

```bash
quasar build --debug
```

In debug mode, account validation failures produce descriptive log messages like:

```
Account 'escrow' (index 1): must be writable, no duplicates
```

Without debug mode, these log statements are compiled out entirely, saving compute units and binary size. Always build without `--debug` for production deployments.

## The `#![no_std]` Requirement

Quasar programs are `#![no_std]` by design. The `prelude::*` import provides:

- `ProgramError`, `Address`, `AccountView` -- core Solana types
- `address!`, `declare_id!` -- address construction macros
- `Ctx`, `CtxWithRemaining`, `Context` -- instruction context types
- `Account`, `Signer`, `UncheckedAccount`, `SystemAccount`, `Program`, `Interface`, `Sysvar` -- account wrappers
- `System`, `SYSTEM_PROGRAM_ID` -- system program marker type and address
- `Rent`, `Clock` -- sysvar types
- `String`, `Vec`, `RawEncoded` -- dynamic field marker types
- `PodU64`, `PodU32`, `PodU16`, `PodI64`, etc. -- alignment-1 integer types
- `ParseAccounts`, `AccountCount`, `AsAccountView`, `Discriminator`, `Owner`, `Space`, `Id`, `Event` -- framework traits
- `Deref`, `DerefMut` -- re-exported from `core::ops`
- `emit!`, `require!`, `require_eq!`, `require_keys_eq!` -- utility macros
- `log` -- logging function
- `Seed` -- CPI seed type

## Instruction Handler Pattern

Each handler accepts `ctx: Ctx<T>` (or `ctx: CtxWithRemaining<T>`) as its first parameter, plus any additional typed arguments, and returns `Result<(), ProgramError>`. Business logic goes in `impl` blocks on the accounts struct:

```rust
#[instruction(discriminator = 0)]
pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
    ctx.accounts.deposit(amount)
}
```

For instructions that forward a variable number of accounts to CPIs, use `CtxWithRemaining`:

```rust
#[instruction(discriminator = 0)]
pub fn create(ctx: CtxWithRemaining<Create>, threshold: u8) -> Result<(), ProgramError> {
    ctx.accounts.create_multisig(threshold, &ctx.bumps, ctx.remaining_accounts())
}
```
