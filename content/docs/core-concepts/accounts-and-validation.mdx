---
title: Accounts and validation
description: How Quasar turns the SVM account list into validated, zero-copy views before a handler runs.
---

An instruction enters a Solana program with an ordered list of accounts and an
instruction-data buffer. Quasar's `#[derive(Accounts)]` turns that untyped input
into a context whose fields have already passed the checks declared by the
program.

The central rule is simple: **the handler should receive capabilities, not raw
accounts**. A `Signer` has signed, an `Account<Vault>` has the expected owner
and layout, and a `Program<SystemProgram>` is the executable program at the
expected address.

<Callout type="warn" title="Validation is part of the trust boundary">
Quasar generates the checks you declare; it cannot infer your authority model.
Choosing `UncheckedAccount`, omitting `has_one`, or accepting the wrong owner is
still a program bug even when parsing succeeds.
</Callout>

## The context is executable policy

An accounts struct puts the rules for one instruction in one place:

```rust
#[derive(Accounts)]
pub struct Withdraw {
    #[account(mut, has_one(authority), has_one(mint))]
    pub vault: Account<Vault>,
    pub authority: Signer,
    pub mint: InterfaceAccount<Mint>,
    pub token_program: Interface<TokenInterface>,
}
```

Before `Withdraw` is constructed, Quasar establishes that:

- `vault` is writable, owned by this program, and has `Vault`'s discriminator
  and data layout;
- the stored `vault.authority` and `vault.mint` addresses match the accounts in
  this instruction;
- `authority` signed the transaction;
- `mint` is a valid token mint owned by a supported token program; and
- `token_program` is an executable program accepted by `TokenInterface`.

The handler can then focus on the state transition. For the complete list of
wrappers and their exact checks, use the [account types
reference](/docs/references/account-types). For every field attribute and error,
use the [account constraints reference](/docs/references/account-constraints).

## Validation order

For each instruction, generated code performs these stages:

1. **Header checks.** Signer, writable, executable, and duplicate expectations
   are compared with the runtime account metadata.
2. **Typed construction.** Each wrapper validates the owner, address,
   discriminator, size, and layout required by its type.
3. **PDA and initialization work.** Typed address expressions are verified;
   accounts declared with `init` are created when appropriate.
4. **Field constraints.** `has_one`, `constraints(...)`, and explicit `address`
   checks run in declaration order.
5. **Handler execution.** Only a fully constructed context reaches your
   instruction body.
6. **Epilogue work.** Declared `close` operations run after a successful
   handler.

This order matters. For example, a business constraint can read `vault.balance`
because the owner and `Vault` layout were checked when `Account<Vault>` was
constructed.

## Inspect the generated plan

The compiler writes its resolved account mechanics into the IDL under the
versioned `quasar:validationPlan` extension. Print the review-oriented form
with:

```bash
quasar build
quasar audit
```

The report lists each account's access flags, wrapper, load mode, and ordered
work before load, after load, and after the handler. It also shows rent-fetch
behavior for instructions that initialize or resize accounts.

Use JSON when a review tool needs the same data:

```bash
quasar audit --json > validation-plan.json
```

The plan answers “what code did the compiler generate?” It cannot answer
whether the declared checks match the protocol's intended authority model.
Review both the accounts struct and the plan.

## Account data stays borrowed

`Account<T>` does not deserialize account bytes into a separate Rust value. The
`#[account]` macro generates an alignment-one companion layout and exposes a
typed view over the validated account region.

```rust
#[account(discriminator = 1, set_inner)]
#[seeds(b"vault", authority: Address)]
pub struct Vault {
    pub authority: Address,
    pub balance: u64,
    pub bump: u8,
    pub _reserved: [u8; 32],
}
```

Integer fields use Quasar's alignment-one Pod representations internally.
Normal arithmetic and comparisons still work, but release builds use wrapping
integer semantics. Use checked arithmetic when overflow must reject the
instruction.

Dynamic `String<MAX>` and `Vec<T, MAX>` fields are also borrowed after their
header and payload ranges are validated. Mutation goes through a guard so the
account can be resized and rent-adjusted when the guard saves.

The [account layout guide](/docs/zero-copy/account-layout) covers discriminators,
versions, reserved bytes, and upgrade-safe layouts. [Dynamic
fields](/docs/zero-copy/dynamic-fields) covers bounded strings, vectors, and
mutation guards.

## Mutability and duplicate addresses are explicit

Fields are read-only unless the context declares `#[account(mut)]` or an
operation such as `init` requires writability. If the transaction supplies a
read-only account for a mutable field, parsing fails before the handler runs.

Quasar also rejects duplicate addresses by default. This prevents a caller
from satisfying two logical roles with the same account and lets the generated
entrypoint avoid general borrow-state tracking.

When one address intentionally fills two roles, opt in on the later field and
record the invariant:

```rust
/// CHECK: The same authority intentionally signs both sides of this operation.
#[account(dup)]
pub authority_alias: Signer,
```

Treat `dup` like `UncheckedAccount`: it is a review point, not a convenience
flag.

## Put address derivation next to the account type

Typed seeds define a PDA once:

```rust
#[account(discriminator = 1)]
#[seeds(b"vault", authority: Address)]
pub struct Vault {
    pub authority: Address,
    pub bump: u8,
}
```

Instruction contexts bind fields to that definition:

```rust
#[account(
    init,
    payer = authority,
    address = Vault::seeds(authority.address())
)]
pub vault: Account<Vault>,
```

Initialization discovers the canonical bump and exposes it through
`ctx.bumps.vault`. Store that bump in the account when later instructions need
the faster stored-bump verification path. The generated `vault_signer` helper
reuses the same seed definition for signed CPIs.

See [Program-derived addresses](/docs/core-concepts/pda) for typed seed rules,
verification paths, and signer helpers.

## Choose constraints by responsibility

- Use the **wrapper type** for structural identity: owner, executable program,
  sysvar address, discriminator, and layout.
- Use **`address`** for a canonical key or a typed PDA.
- Use **`has_one`** when stored state names another account in the context.
- Use **`constraints(...)`** for business invariants that depend on validated
  values.
- Use **`init`, `realloc`, and `close`** for account lifecycle operations whose
  payer and destination are explicit in the context.

Custom errors belong on the constraint that can fail:

```rust
#[account(
    mut,
    has_one(authority) @ VaultError::WrongAuthority,
    constraints(vault.balance >= amount) @ VaultError::InsufficientBalance,
)]
pub vault: Account<Vault>,
```

This keeps failures attributable to the policy that rejected the account.

## When more than one owner is valid

`Interface<T>` validates a program against a set of accepted program IDs.
`InterfaceAccount<T>` applies the same idea to data accounts, then exposes the
layout supported by the interface. Quasar's token interface uses this to accept
SPL Token and Token-2022 without weakening the owner check to “anything.”

For protocols whose owners use different layouts, `resolve()` dispatches on
the owner after the initial interface check. The result is still a borrowed
view; the dispatch does not deserialize or allocate.
