The `#[instruction]` macro generates discriminator validation, argument deserialization, and context construction for each handler.

## Defining an Instruction

Each instruction is a function annotated with `#[instruction(discriminator = N)]` inside a `#[program]` module:

```rust
#[program]
mod quasar_vault {
    use super::*;

    #[instruction(discriminator = 0)]
    pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.deposit(amount)
    }

    #[instruction(discriminator = 1)]
    pub fn withdraw(ctx: Ctx<Withdraw>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.withdraw(amount)
    }
}
```

Each instruction function must:

1. Have `ctx: Ctx<T>` or `ctx: CtxWithRemaining<T>` as its first parameter, where `T` implements `ParseAccounts + AccountCount`.
2. Specify a `discriminator` value in the attribute.
3. Return `Result<(), ProgramError>` (or `Result<T, ProgramError>` for return data).

## Discriminators

Quasar uses developer-specified integer discriminators, not hash-derived ones. The wire format is fully predictable.

```rust
#[instruction(discriminator = 0)]    // instruction data starts with [0x00]
pub fn deposit(...) -> Result<(), ProgramError> { ... }

#[instruction(discriminator = 1)]    // instruction data starts with [0x01]
pub fn withdraw(...) -> Result<(), ProgramError> { ... }
```

### Discriminator Rules

- **Same length**: All discriminators in a program must have the same byte length. Single-byte gives you 255 possible instructions (0x00 through 0xFE).
- **No duplicates**: Checked at compile time.
- **No 0xFF prefix**: Reserved for the event self-CPI protocol. Compile error if violated.
- **Multi-byte**: For programs needing more than 255 instructions: `discriminator = [0, 1]`.

### Dispatch Routing

The generated `__dispatch` function checks the first byte for `0xFF` (routes to event handler), then reads the discriminator prefix and matches against the dispatch table. Unmatched discriminators return `ProgramError::InvalidInstructionData`.

## Argument Deserialization

After the discriminator prefix, the remaining instruction data contains the function arguments. Fixed-size types use zero-copy pointer casts; dynamic types use inline prefix reads.

### Fixed-Size Arguments

Fixed-size arguments (`u64`, `Address`, `bool`, etc.) are packed into a `#[repr(C)]` struct with Pod types (alignment 1) and read via pointer cast:

```rust
#[instruction(discriminator = 0)]
pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
    ctx.accounts.deposit(amount)
}
```

Generated code:

```rust
#[repr(C)]
#[derive(Copy, Clone)]
struct InstructionDataZc {
    amount: PodU64,  // alignment 1
}

const _: () = assert!(core::mem::align_of::<InstructionDataZc>() == 1);

let __zc = unsafe { &*(ctx.data.as_ptr() as *const InstructionDataZc) };
let amount = __zc.amount.get();  // PodU64::get() -> u64
```

### Dynamic Arguments

Dynamic arguments use length-prefixed encoding. `String<P, MAX>` (where `P` is the prefix type: `u8`, `u16`, or `u32`) and `Vec<T, P, MAX>`:

```rust
#[instruction(discriminator = 2)]
pub fn set_label(ctx: Ctx<SetLabel>, label: String<32>) -> Result<(), ProgramError> {
    ctx.accounts.update_label(label)
}
```

`String<32>` uses the default `u32` prefix. The generated deserialization:
1. Reads the length prefix bytes (1, 2, or 4 depending on prefix type).
2. Validates the length does not exceed `MAX`.
3. Validates UTF-8 encoding.
4. Produces a `&str` reference pointing directly into the instruction data buffer.

For `Vec<T, P, MAX>`, the process is similar but produces a `&[T]` slice. Vec element types must have alignment 1, enforced by a compile-time assertion.

### Tail Arguments

A tail argument consumes all remaining instruction data after fixed and prefixed fields. Use bare `&[u8]` or `&str` as the parameter type:

```rust
#[instruction(discriminator = 0)]
pub fn process(ctx: Ctx<Process>, data: &[u8]) -> Result<(), ProgramError> {
    // data is all remaining instruction bytes after fixed fields
}
```

Only one tail argument is allowed per instruction, and it must be the last parameter. `&str` tail arguments additionally validate UTF-8 encoding.

### Wire Format Summary

The instruction data layout for `deposit(ctx, amount: u64)` with `discriminator = 0`:

```text
[0x00]                    <- discriminator (1 byte)
[amount: 8 bytes LE]      <- u64 argument
```

For `set_label(ctx, label: String<32>)` with `discriminator = 2` (default `u32` prefix):

```text
[0x02]                         <- discriminator (1 byte)
[len: 4 bytes LE]              <- u32 string length prefix
[label: len bytes UTF-8]       <- string data
```

## Context Types

### `Ctx<'info, T>`

```rust
pub struct Ctx<'info, T: ParseAccounts<'info> + AccountCount> {
    pub accounts: T,
    pub bumps: T::Bumps,
    pub program_id: &'info [u8; 32],
    pub data: &'info [u8],        // instruction data, discriminator already consumed
}
```

### `CtxWithRemaining<'info, T>`

Same fields as `Ctx`, plus a `remaining_accounts()` method that returns a `RemainingAccounts` iterator over accounts past the declared set:

```rust
pub fn create(ctx: CtxWithRemaining<Create>, threshold: u8) -> Result<(), ProgramError> {
    ctx.accounts.create_multisig(threshold, &ctx.bumps, ctx.remaining_accounts())
}
```

## Return Data

Instructions can return data to the caller via `sol_set_return_data`. Use `Result<T, ProgramError>` where `T` is a fixed-size type with alignment 1:

```rust
#[instruction(discriminator = 0)]
pub fn compute_value(ctx: Ctx<Compute>) -> Result<PodU64, ProgramError> {
    Ok(PodU64::from(42u64))
}
```

The generated code runs the epilogue (close operations) before setting return data, then rewrites the function signature to return `Result<(), ProgramError>`. A compile-time assertion enforces alignment 1 on the return type.

## Instruction Data Access in Account Validation

The `#[instruction(...)]` attribute on an accounts struct allows access to instruction arguments during account parsing:

```rust
#[derive(Accounts)]
#[instruction(name: String<32>, symbol: String<10>)]
pub struct CreateNft<'info> {
    #[account(init, payer = authority, metadata::name = name, metadata::symbol = symbol)]
    pub metadata: &'info mut Account<Metadata>,
    pub authority: &'info mut Signer,
}
```

## Epilogue

After the instruction handler returns `Ok(())`, the generated code calls `accounts.epilogue()`, which executes `close` operations for any accounts marked with `close = destination`. The handler's result is checked first -- if it returns an error, the epilogue is skipped.
