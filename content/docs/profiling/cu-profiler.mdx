The Quasar CU profiler measures compute unit usage by statically walking every instruction in the `.text` section of your compiled sBPF binary. Unlike runtime profiling, this approach is deterministic and does not require executing the program on a validator.

## How It Works

The profiler:

1. Parses the ELF binary to extract the `.text` section and symbol table
2. Walks every instruction in the `.text` section, counting each as one compute unit
3. For each instruction address, resolves the function name via DWARF debug info (with inline frame support) or falls back to symbol table lookup
4. Aggregates instruction counts into folded stacks (for flamegraph generation) and a per-function leaf count table
5. Outputs a per-function CU breakdown sorted by cost, with optional delta tracking against the previous run

Each sBPF instruction costs one compute unit, so the instruction count directly corresponds to CU usage. The `lddw` instruction (opcode `0x18`) is a special case -- it occupies two 8-byte slots (16 bytes total) but counts as a single instruction.

## Running the Profiler

```bash
quasar profile
```

This automatically runs a profile build (with debug symbols) if needed, then profiles the resulting binary. You can also pass an explicit ELF path:

```bash
quasar profile target/deploy/my_program.so
```

The profiler requires debug symbols. If you pass a fully stripped binary, it exits with an error directing you to use the unstripped binary from `target/sbf-solana-solana/release/` instead of `target/deploy/`. You can also build with `quasar build --debug`.

## Reading the Output

On the first run (no baseline exists), the profiler shows the top 5 hottest functions:

```
  my_program  2,816 CU
     1,200  42.6%  Initialize::verify
       800  28.4%  Deposit::process
       400  14.2%  entrypoint
       300  10.6%  Withdraw::process
       116   4.1%  +3 more (--expand)
```

Each line shows:
- **CU count** -- Number of compute units attributed to that function
- **Percentage** -- Share of total CU budget
- **Function name** -- Simplified from the demangled Rust symbol

## Tracking Changes Between Runs

The profiler saves a baseline after each run to `target/profile/.last-profile.<program>` (a simple text file mapping function names to CU counts). On subsequent runs, it shows the biggest regressions and improvements sorted by absolute magnitude:

```
  my_program  2,816 CU (+42)
     1,200 (+200)   5.0%  Initialize::verify
       800 (-158)   3.0%  Deposit::process
```

- Red `(+N)` indicates a regression (more CU)
- Green `(-N)` indicates an improvement (fewer CU)
- The total delta appears next to the program name

This makes it easy to catch CU regressions as you iterate on your program.

## `--diff`: Compare Against On-Chain Programs

Open the diff view for comparing profiles:

```bash
quasar profile --diff <PROGRAM_NAME>
```

This prints a flamegraph URL with `&view=diff` and starts the profiler server in blocking mode (foreground). Open the URL in your browser to see the diff view. Press Ctrl-C to stop the server when done.

## `--watch`: Continuous Profiling

Re-profile automatically whenever source files change:

```bash
quasar profile --watch
```

Combined with `--expand`, this gives you a live CU dashboard as you optimize:

```bash
quasar profile --watch --expand
```

## `--share`: Publish to GitHub Gist

Upload the profile JSON as a public GitHub Gist:

```bash
quasar profile --share
```

This requires the [GitHub CLI](https://cli.github.com/) (`gh`) to be installed and authenticated (`gh auth login`). The profile JSON is uploaded via `gh gist create --public` and a shareable URL is printed. The `--share` flag skips starting the local flamegraph server.

## Profile Data Format

Each profile run generates a timestamped JSON file in `target/profile/profiles/`:

```
target/profile/profiles/my_program__2025-03-15-14-30-00-123.profile.json
```

The double underscore separates the program name from the UTC timestamp. The JSON contains:

- Program name and version (resolved from the workspace's `Cargo.toml`)
- Binary hash (SHA-256) and size in bytes
- A hierarchical frame tree built from folded stacks, suitable for flamegraph rendering

This data is consumed by the [interactive flamegraph viewer](/docs/profiling/flamegraphs).

## Tips for Reducing CU

- **Inline aggressively** -- Use `#[inline(always)]` on instruction handlers. Quasar does this by default in generated scaffolding.
- **Avoid unnecessary checks** -- Every branch in your program adds instructions. Quasar's zero-copy validation is already minimal.
- **Minimize CPI** -- Cross-program invocations are expensive. The `system_program.transfer()` helper is optimized, but each CPI call still adds overhead.
- **Use `--expand`** -- Find the hottest functions and focus optimization there.
- **Profile early and often** -- Use `--watch` during development to catch regressions immediately.
