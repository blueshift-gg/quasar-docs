title: Rust Client
description: Using the auto-generated Rust client for off-chain instruction building and testing.

Quasar generates Rust client code in two forms: an inline `client` module inside your program crate (compiled only for non-SBF targets), and a standalone crate at `target/client/rust/<name>-client/`. Both provide typed instruction builders that produce `solana_instruction::Instruction` values, ready to use with `solana-client`, Mollusk, or any Solana test harness.

## The inline client module

The `#[program]` macro generates a `client` submodule inside the program module, then re-exports it at the crate root. This module is gated behind `#[cfg(not(any(target_arch = "bpf", target_os = "solana")))]`, so it only compiles for off-chain targets (tests, CLI tools, scripts).

For each instruction, the client module contains a struct that implements `From<T> for Instruction`. The struct is generated by a `macro_rules!` bridge emitted by `#[derive(Accounts)]` and invoked by `#[program]`. Here is the expanded output for the vault program:

```rust
// Generated by #[program] + #[derive(Accounts)]
pub mod client {
    use super::*;

    pub struct DepositInstruction {
        pub user: Address,
        pub vault: Address,
        pub system_program: Address,
        pub amount: u64,
    }

    impl From<DepositInstruction> for quasar_lang::client::Instruction {
        fn from(ix: DepositInstruction) -> quasar_lang::client::Instruction {
            let accounts = vec![
                quasar_lang::client::AccountMeta::new(ix.user, true),       // mut Signer
                quasar_lang::client::AccountMeta::new(ix.vault, false),     // mut
                quasar_lang::client::AccountMeta::new_readonly(ix.system_program, false),
            ];
            let data = quasar_lang::client::build_instruction_data(
                &[0], // discriminator
                |_data| {
                    quasar_lang::client::WriteBytes::write_bytes(&ix.amount, _data);
                },
            );
            quasar_lang::client::Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }

    pub struct WithdrawInstruction {
        pub user: Address,
        pub vault: Address,
        pub amount: u64,
    }

    impl From<WithdrawInstruction> for quasar_lang::client::Instruction {
        fn from(ix: WithdrawInstruction) -> quasar_lang::client::Instruction {
            let accounts = vec![
                quasar_lang::client::AccountMeta::new(ix.user, true),
                quasar_lang::client::AccountMeta::new(ix.vault, false),
            ];
            let data = quasar_lang::client::build_instruction_data(
                &[1], // discriminator
                |_data| {
                    quasar_lang::client::WriteBytes::write_bytes(&ix.amount, _data);
                },
            );
            quasar_lang::client::Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }
}
```

The struct names follow the pattern `<PascalCaseInstructionName>Instruction`. Each struct has a field for every account in the corresponding `#[derive(Accounts)]` struct (as `Address` values) plus every instruction argument. The `#[program]` macro re-exports the client module at the crate root, so consumers import it as `my_program::client::*`.

### Account metadata

The `AccountMeta` for each field is derived automatically from the accounts struct:

- **Writable**: `&'info mut T` references or explicit `#[account(mut)]` attributes produce `AccountMeta::new()`.
- **Signer**: The `Signer` type produces `is_signer = true`.
- **Read-only, non-signer**: All other accounts produce `AccountMeta::new_readonly()` with `is_signer = false`.

The codegen checks both the type-level reference mutability and the `#[account(mut)]` attribute -- either one is sufficient to mark an account writable.

### Instruction data

The instruction data is built by `build_instruction_data`, which writes the discriminator bytes first, then invokes a closure that serializes each argument in order via the `WriteBytes` trait. The supported types are:

| Rust type | Encoding |
|-----------|----------|
| `u8`, `u16`, `u32`, `u64`, `u128` | Little-endian bytes |
| `i8`, `i16`, `i32`, `i64`, `i128` | Little-endian bytes |
| `bool` | Single byte (0 or 1) |
| `Address` | 32 bytes |
| `[u8; N]` | Raw N bytes |
| `Vec<T>` | u32 LE length prefix + serialized elements |
| `TailBytes` | Raw bytes appended with no length prefix |

### Remaining accounts

If your instruction uses `CtxWithRemaining<T>` instead of `Ctx<T>`, the generated struct includes an additional `remaining_accounts: Vec<AccountMeta>` field that gets appended after the fixed accounts.

## Using the inline client in tests

The typical use of the inline client is testing with Mollusk or other SVM test harnesses. Here is the actual deposit test from the vault example (`examples/vault/src/tests.rs`):

```rust
extern crate std;

use {
    crate::idl_client::{DepositInstruction, WithdrawInstruction},
    mollusk_svm::{program::keyed_account_for_system_program, Mollusk},
    quasar_lang::prelude::*,
    solana_account::Account,
    solana_instruction::Instruction,
};

fn setup() -> Mollusk {
    Mollusk::new(&crate::ID, "../../target/deploy/quasar_vault")
}

#[test]
fn test_deposit() {
    let mollusk = setup();

    let (system_program, system_program_account) = keyed_account_for_system_program();

    let user = Address::new_unique();
    let user_account = Account::new(10_000_000_000, 0, &system_program);

    let (vault, _vault_bump) =
        Address::find_program_address(&[b"vault", user.as_ref()], &crate::ID);
    let vault_account = Account::new(0, 0, &system_program);

    let deposit_amount: u64 = 1_000_000_000;

    let instruction: Instruction = DepositInstruction {
        user,
        vault,
        system_program,
        amount: deposit_amount,
    }
    .into();

    let result = mollusk.process_instruction(
        &instruction,
        &[
            (user, user_account.clone()),
            (vault, vault_account.clone()),
            (system_program, system_program_account.clone()),
        ],
    );

    assert!(
        result.program_result.is_ok(),
        "deposit failed: {:?}",
        result.program_result
    );

    let user_after = result.resulting_accounts[0].1.lamports;
    let vault_after = result.resulting_accounts[1].1.lamports;

    assert_eq!(user_after, 10_000_000_000 - deposit_amount);
    assert_eq!(vault_after, deposit_amount);
}
```

The pattern: construct the instruction struct with addresses and arguments, convert it to an `Instruction` with `.into()`, and pass it to the test harness.

Note that the vault example imports from `crate::idl_client` (a manually written client module) rather than from the auto-generated `crate::client` module. For programs that use the generated client, the import would be `use crate::client::*;` or from an external test crate, `use my_program::client::*;`.

## The standalone client crate

Running `quasar idl` also generates a standalone Rust crate at:

```
target/
  client/
    rust/
      <name>-client/
        Cargo.toml
        src/
          lib.rs
```

This crate contains the same instruction builder structs but as a standalone library that can be added as a dependency by other crates. It is useful when you have a separate CLI tool or integration test suite that lives outside the program crate.

The generated `Cargo.toml` includes the necessary dependencies (versions are read from your program's `Cargo.toml`):

```toml
[package]
name = "<name>-client"
version = "<version>"
edition = "2021"

[dependencies]
solana-instruction = "..."
solana-address = "..."
```

The `generate_client` function in `quasar-idl` produces the same struct/impl pattern as the inline macro but as standalone Rust source that does not depend on your program crate.

## Integration with solana-client

For deploying transactions to a live cluster or localnet, combine the generated instructions with `solana-client`:

```rust
use {
    solana_client::rpc_client::RpcClient,
    solana_sdk::{
        signature::{Keypair, Signer},
        transaction::Transaction,
    },
    solana_instruction::Instruction,
};

let client = RpcClient::new("http://localhost:8899");
let payer = Keypair::new();

// Build the instruction
let ix: Instruction = DepositInstruction {
    user: payer.pubkey(),
    vault: vault_pda,
    system_program: solana_sdk::system_program::id(),
    amount: 1_000_000_000,
}.into();

// Build and send the transaction
let recent_blockhash = client.get_latest_blockhash()?;
let tx = Transaction::new_signed_with_payer(
    &[ix],
    Some(&payer.pubkey()),
    &[&payer],
    recent_blockhash,
);
client.send_and_confirm_transaction(&tx)?;
```

## How code generation works

The client code generation happens at two levels:

### Compile-time (proc-macro)

`#[derive(Accounts)]` emits a hidden `macro_rules!` bridge named `__<snake_name>_instruction` for each accounts struct. This bridge captures account field names and their `AccountMeta` properties (writable, signer). It accepts three parameters: a struct name, discriminator bytes, and argument names/types.

`#[program]` scans all `#[instruction]` functions, extracts the accounts type and arguments from each function signature, and invokes the corresponding bridge macro. The results are collected into a `client` submodule inside the program module, then re-exported at the crate root with `pub use <mod_name>::client;`. This happens during compilation, so the client code is type-checked alongside your program.

### CLI-time (quasar idl)

`quasar idl` parses your program source at the AST level (using `syn` via the `quasar-idl` crate), extracts the same information, and writes both the standalone Rust crate and the TypeScript clients. The standalone crate does not depend on your program code -- it only depends on `solana-instruction` and `solana-address`.

Both paths produce identical instruction encodings. The inline module is most convenient for tests within the program crate; the standalone crate is better for external consumers.

## The WriteBytes trait

The instruction data serialization is powered by the `WriteBytes` trait in `quasar_lang::client`. This trait defines a single method:

```rust
pub trait WriteBytes {
    fn write_bytes(&self, buf: &mut Vec<u8>);
}
```

It is implemented for all integer types (little-endian), `bool`, `Address` (32 bytes), fixed arrays `[u8; N]`, `Vec<T>` (u32 length prefix + elements), and `TailBytes` (raw bytes with no length prefix). The `build_instruction_data` helper combines the discriminator bytes with the argument serialization:

```rust
let data = build_instruction_data(
    &[0], // discriminator
    |data| {
        amount.write_bytes(data);
    },
);
```

This is the only module in `quasar-lang` that allocates (using `Vec<u8>`), since off-chain code runs with a standard allocator.

## Custom client code

If the generated client does not cover your needs, you can write a manual client module. The vault example does this with `src/idl_client.rs`, which mirrors the generated client but is hand-written:

```rust
// src/idl_client.rs
use {
    alloc::vec,
    quasar_lang::prelude::*,
    solana_instruction::{AccountMeta, Instruction},
};

pub struct DepositInstruction {
    pub user: Address,
    pub vault: Address,
    pub system_program: Address,
    pub amount: u64,
}

impl From<DepositInstruction> for Instruction {
    fn from(ix: DepositInstruction) -> Instruction {
        let accounts = vec![
            AccountMeta::new(ix.user, true),
            AccountMeta::new(ix.vault, false),
            AccountMeta::new_readonly(ix.system_program, false),
        ];
        let mut data = vec![0];
        data.extend_from_slice(&ix.amount.to_le_bytes());
        Instruction {
            program_id: crate::ID,
            accounts,
            data,
        }
    }
}
```

The vault includes it conditionally:

```rust
#[cfg(test)]
mod idl_client;
#[cfg(test)]
mod tests;
```

This approach is useful during development before the IDL pipeline is integrated, or when you need custom logic (PDA derivation, batched instruction building) that the code generator does not handle.
