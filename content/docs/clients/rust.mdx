title: Rust Client
description: Using the auto-generated Rust client for off-chain instruction building and testing.

Quasar generates Rust client code in two forms: an inline `client` module inside your program crate (compiled only for non-SBF targets), and a standalone crate at `target/client/rust/<name>-client/`. Both provide typed instruction builders that produce `solana_instruction::Instruction` values, ready to use with `solana-client`, Mollusk, or any Solana test harness.

## The inline client module

When you define a `#[program]` module, Quasar generates a `client` submodule inside it. This module is gated behind `#[cfg(not(any(target_arch = "bpf", target_os = "solana")))]`, so it only compiles for off-chain targets (tests, CLI tools, scripts).

For each instruction in your program, the client module contains a struct that implements `From<T> for Instruction`. Here is what the generated client looks like for the vault program:

```rust
// Generated by #[program] + #[derive(Accounts)]
pub mod client {
    use super::*;

    pub struct DepositInstruction {
        pub user: Address,
        pub vault: Address,
        pub system_program: Address,
        pub amount: u64,
    }

    impl From<DepositInstruction> for Instruction {
        fn from(ix: DepositInstruction) -> Instruction {
            let accounts = vec![
                AccountMeta::new(ix.user, true),       // mut Signer
                AccountMeta::new(ix.vault, false),      // mut
                AccountMeta::new_readonly(ix.system_program, false),
            ];
            let mut data = vec![0]; // discriminator
            data.extend_from_slice(&ix.amount.to_le_bytes());
            Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }

    pub struct WithdrawInstruction {
        pub user: Address,
        pub vault: Address,
        pub amount: u64,
    }

    impl From<WithdrawInstruction> for Instruction {
        fn from(ix: WithdrawInstruction) -> Instruction {
            let accounts = vec![
                AccountMeta::new(ix.user, true),
                AccountMeta::new(ix.vault, false),
            ];
            let mut data = vec![1]; // discriminator
            data.extend_from_slice(&ix.amount.to_le_bytes());
            Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }
}
```

The struct names follow the pattern `<PascalCaseInstructionName>Instruction`. Each struct has a field for every account in the corresponding `#[derive(Accounts)]` struct (as `Address` values) plus every instruction argument.

### Account metadata

The `AccountMeta` for each field is derived automatically from the accounts struct:

- **Writable**: `&'info mut T` or explicit `#[account(mut)]` produces `AccountMeta::new()`.
- **Signer**: `Signer` type produces `is_signer = true`.
- **Read-only, non-signer**: All other accounts produce `AccountMeta::new_readonly()` with `is_signer = false`.

### Instruction data

The instruction data is built by writing the discriminator bytes first, then serializing each argument in order using the `WriteBytes` trait. The supported types are:

| Rust type | Encoding |
|-----------|----------|
| `u8`, `u16`, `u32`, `u64`, `u128` | Little-endian bytes |
| `i8`, `i16`, `i32`, `i64`, `i128` | Little-endian bytes |
| `bool` | Single byte (0 or 1) |
| `Address` | 32 bytes |
| `[u8; N]` | Raw N bytes |
| `Vec<T>` | u32 LE length prefix + serialized elements |

### Remaining accounts

If your instruction uses `CtxWithRemaining<T>` instead of `Ctx<T>`, the generated struct includes an additional `remaining_accounts: Vec<AccountMeta>` field that gets appended after the fixed accounts.

## Using the inline client in tests

The most common use case for the inline client is testing with Mollusk or other SVM test harnesses. Here is a complete test from the vault example:

```rust
use {
    crate::idl_client::{DepositInstruction, WithdrawInstruction},
    mollusk_svm::{program::keyed_account_for_system_program, Mollusk},
    quasar_lang::prelude::*,
    solana_account::Account,
    solana_instruction::Instruction,
};

fn setup() -> Mollusk {
    Mollusk::new(&crate::ID, "../../target/deploy/quasar_vault")
}

#[test]
fn test_deposit() {
    let mollusk = setup();

    let (system_program, system_program_account) = keyed_account_for_system_program();

    let user = Address::new_unique();
    let user_account = Account::new(10_000_000_000, 0, &system_program);

    let (vault, _bump) =
        Address::find_program_address(&[b"vault", user.as_ref()], &crate::ID);
    let vault_account = Account::new(0, 0, &system_program);

    let instruction: Instruction = DepositInstruction {
        user,
        vault,
        system_program,
        amount: 1_000_000_000,
    }
    .into();

    let result = mollusk.process_instruction(
        &instruction,
        &[
            (user, user_account),
            (vault, vault_account),
            (system_program, system_program_account),
        ],
    );

    assert!(result.program_result.is_ok());

    let vault_after = result.resulting_accounts[1].1.lamports;
    assert_eq!(vault_after, 1_000_000_000);
}
```

The pattern is straightforward: construct the instruction struct with addresses and arguments, convert it to an `Instruction` with `.into()`, and pass it to the test harness.

## The standalone client crate

Running `quasar idl` also generates a standalone Rust crate at:

```
target/
  client/
    rust/
      <name>-client/
        Cargo.toml
        src/
          lib.rs
```

This crate contains the same instruction builder structs but as a standalone library that can be added as a dependency by other crates. It is useful when you have a separate CLI tool or integration test suite that lives outside the program crate.

The generated `Cargo.toml` includes the necessary dependencies:

```toml
[package]
name = "<name>-client"
version = "<version>"
edition = "2021"

[dependencies]
solana-instruction = "..."
solana-address = "..."
```

## Integration with solana-client

For deploying transactions to a live cluster or localnet, combine the generated instructions with `solana-client`:

```rust
use {
    solana_client::rpc_client::RpcClient,
    solana_sdk::{
        signature::{Keypair, Signer},
        transaction::Transaction,
    },
    solana_instruction::Instruction,
};

let client = RpcClient::new("http://localhost:8899");
let payer = Keypair::new();

// Build the instruction
let ix: Instruction = DepositInstruction {
    user: payer.pubkey(),
    vault: vault_pda,
    system_program: solana_sdk::system_program::id(),
    amount: 1_000_000_000,
}.into();

// Build and send the transaction
let recent_blockhash = client.get_latest_blockhash()?;
let tx = Transaction::new_signed_with_payer(
    &[ix],
    Some(&payer.pubkey()),
    &[&payer],
    recent_blockhash,
);
client.send_and_confirm_transaction(&tx)?;
```

## How code generation works

The client code generation happens at two levels:

### Compile-time (proc-macro)

The `#[derive(Accounts)]` macro emits a hidden `macro_rules!` bridge for each accounts struct. This bridge captures the account field names, their `AccountMeta` properties (writable, signer), and makes them available for the `#[program]` macro to consume.

The `#[program]` macro then iterates over all `#[instruction]` functions, invokes the corresponding bridge macro for each, and collects the results into the `client` module. This happens during compilation, so the client code is type-checked alongside your program.

### CLI-time (quasar idl)

The `quasar idl` command parses your program source at the AST level (using `syn`), extracts the same information, and generates the standalone crate. This produces an independent crate that does not depend on your program code -- it only depends on `solana-instruction` and `solana-address`.

Both paths produce identical instruction encodings. The inline module is most convenient for tests within the program crate; the standalone crate is better for external consumers.

## The WriteBytes trait

The instruction data serialization is powered by the `WriteBytes` trait in `quasar_lang::client`. This trait defines a single method:

```rust
pub trait WriteBytes {
    fn write_bytes(&self, buf: &mut Vec<u8>);
}
```

It is implemented for all integer types (little-endian), `bool`, `Address` (32 bytes), fixed arrays `[u8; N]`, and `Vec<T>` (u32 length prefix + elements). The `build_instruction_data` helper combines the discriminator bytes with the argument serialization:

```rust
let data = build_instruction_data(
    &[0], // discriminator
    |data| {
        amount.write_bytes(data);
    },
);
```

This is the only module in `quasar-lang` that allocates (using `Vec<u8>`), since off-chain code runs with a standard allocator.

## Custom client code

If the generated client does not cover your needs (for example, if you need to add helper methods or PDA derivation functions), you can write a manual client module alongside the generated one. The vault example demonstrates this pattern with an `idl_client.rs` file:

```rust
// src/idl_client.rs -- manually written, mirrors the generated client

pub struct DepositInstruction {
    pub user: Address,
    pub vault: Address,
    pub system_program: Address,
    pub amount: u64,
}

impl From<DepositInstruction> for Instruction {
    fn from(ix: DepositInstruction) -> Instruction {
        let accounts = vec![
            AccountMeta::new(ix.user, true),
            AccountMeta::new(ix.vault, false),
            AccountMeta::new_readonly(ix.system_program, false),
        ];
        let mut data = vec![0];
        data.extend_from_slice(&ix.amount.to_le_bytes());
        Instruction {
            program_id: crate::ID,
            accounts,
            data,
        }
    }
}
```

This is useful during development before the IDL generation pipeline is integrated, or when you need to add custom logic (like PDA derivation or batched instruction building) that the code generator does not handle.
