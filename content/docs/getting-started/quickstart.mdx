![A Quasar program and its passing local test](/images/quickstart.webp)

This guide takes one path through the project wizard: the Solana toolchain, a
minimal program, and Rust tests powered by QuasarSVM. At the end, the generated
program has compiled to SBF and executed locally without a validator.

<Callout title="Before you start">
Install Quasar and its supported Solana toolchain first. The [installation
guide](/docs/getting-started/installation) pins the CLI, Rust, and Agave
versions used by v0.1.0.
</Callout>

## 1. Create the project

```bash
quasar init my-program --yes --toolchain solana --test-language rust --rust-framework quasar-svm --template minimal
cd my-program
```

The command creates `src/lib.rs`, a program keypair, `Cargo.toml`,
`Quasar.toml`, and a generated Rust test. The program, test, template, and
toolchain choices are explicit; Git setup follows your saved CLI preference.

## 2. Build the program

```bash
quasar build
```

A successful build writes the program binary to
`target/deploy/my_program.so`. The program address in `src/lib.rs` comes from
the generated keypair in the same directory.

## 3. Run the generated test

```bash
quasar test
```

Quasar builds the program, loads the SBF binary into QuasarSVM, and runs the
generated `test_initialize` test. Success is a zero exit status with Cargo's
`test result: ok` summary.

## What the test establishes

The minimal handler accepts one signer account and returns successfully. Its
generated test supplies that account, executes instruction discriminator `0`,
and fails with the program logs if execution does not succeed.

This is a local execution check, not a security claim. It does not prove that a
larger program's account constraints, authority model, or state transitions are
correct.

## Extend the project

Use `quasar add` when you are ready to move beyond the minimal handler:

```bash
quasar add -i transfer -s vault -e access
quasar lint --update-lock
```

The first command adds the three modules and wires them into the program. The
second records their generated discriminators in `quasar.lock.json`; commit
that file so future changes cannot silently alter the wire format.

To run one test or enable a program feature during the test build:

```bash
quasar test -f test_init
quasar test --features debug
```

Deployment is deliberately outside the local first-success path because it
needs a funded signer and a cluster. When the program is ready for devnet:

```bash
quasar deploy --url devnet
```

Before deploying, commit `quasar.lock.json` and review the post-deploy manifest.
The [deployment guide](/docs/guides/deploy-and-verify) covers both steps.
