---
title: Migrating from Anchor
description: Step-by-step guide to migrate an existing Anchor program to Quasar.
---

If you have built Solana programs with Anchor, you already know most of what Quasar does. The two frameworks share the same mental model -- derive macros for accounts, typed contexts for instructions, and constraint-based validation. The differences are in the runtime: Quasar is zero-copy and `no_std`, which means no deserialization, no heap allocation, and significantly lower compute unit costs.

This guide maps every Anchor concept to its Quasar equivalent with side-by-side code.

## What stays the same

A lot. If you are productive with Anchor, you will be productive with Quasar immediately:

- **`#[derive(Accounts)]`** works the same way. You define a struct with typed account fields and attribute constraints.
- **Account constraints** like `seeds`, `bump`, `has_one`, `constraint`, `init`, `init_if_needed`, `payer`, `close`, `mut`, and `address` all exist in Quasar with the same syntax.
- **Error codes** start at 6000 by convention and use `#[error_code]` with `#[msg("...")]` annotations.
- **`require!`, `require_eq!`, `require_keys_eq!`** macros work identically.
- **PDA derivation** uses the same `seeds` and `bump` system. Bumps are accessible via a generated `Bumps` companion struct.
- **Project layout** follows the same convention: `lib.rs` as entrypoint, `instructions/` directory, `state.rs` for accounts, `events.rs` for events.

## Program declaration

**Anchor:**
```rust
use anchor_lang::prelude::*;

declare_id!("11111111111111111111111111111111");

#[program]
mod my_program {
    use super::*;

    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        // ...
        Ok(())
    }
}
```

**Quasar:**
```rust
use quasar_lang::prelude::*;

declare_id!("11111111111111111111111111111111");

#[program]
mod my_program {
    use super::*;

    #[instruction(discriminator = 0)]
    pub fn initialize(ctx: Ctx<Initialize>) -> Result<(), ProgramError> {
        ctx.accounts.initialize()
    }
}
```

Three differences to note:

1. **Explicit discriminators.** Every instruction has `#[instruction(discriminator = N)]` with a developer-specified value. Anchor uses SHA-256 hashes; Quasar uses explicit integers. This makes the dispatch table smaller and the discriminator cheaper to match (a single integer comparison instead of an 8-byte memcmp). Discriminators must be non-zero and unique within a program.

2. **`Ctx<T>` instead of `Context<T>`.** Functionally the same -- it holds your parsed accounts, bumps, and program ID. Quasar also provides `CtxWithRemaining<T>` for instructions that need access to [remaining accounts](/docs/features/remaining-accounts).

3. **`Result<(), ProgramError>` instead of `Result<()>`.** Quasar does not wrap `ProgramError` in its own result type. You return Solana's native `ProgramError` directly.

## Account types

**Anchor:**
```rust
#[account]
pub struct Escrow {
    pub maker: Pubkey,
    pub mint_a: Pubkey,
    pub mint_b: Pubkey,
    pub receive: u64,
    pub bump: u8,
}
```

**Quasar:**
```rust
#[account(discriminator = 1)]
pub struct Escrow {
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub receive: u64,
    pub bump: u8,
}
```

Key differences:

- **Explicit discriminator.** Like instructions, account types require an explicit `discriminator` value. All-zero discriminators are rejected at compile time to prevent uninitialized data from passing validation.
- **`Address` instead of `Pubkey`.** Quasar uses Solana's `Address` type, which is alignment-1 (a transparent wrapper around `[u8; 32]`). Anchor's `Pubkey` has alignment 4, which is incompatible with zero-copy access on unaligned account data. This is a central design decision -- it means field access requires no copying or padding.
- **Zero-copy companion struct.** The `#[account]` macro generates a `#[repr(C)]` companion type (prefixed `Zc`) with Pod fields. When you access an `Account<Escrow>`, the data is read directly from account memory through pointer casts. There is no Borsh deserialization step.
- **`set_inner()` instead of field assignment.** To write to an account, you call the generated `set_inner()` method which writes all fields at once via the zero-copy companion:

```rust
self.escrow.set_inner(
    *self.maker.address(),
    *self.mint_a.address(),
    *self.mint_b.address(),
    *self.maker_ta_b.address(),
    receive,
    bumps.escrow,
);
```

## Accounts struct and constraints

**Anchor:**
```rust
#[derive(Accounts)]
pub struct Take<'info> {
    #[account(mut)]
    pub taker: Signer<'info>,
    #[account(
        mut,
        has_one = maker,
        constraint = escrow.receive > 0,
        close = taker,
        seeds = [b"escrow", maker.key().as_ref()],
        bump = escrow.bump
    )]
    pub escrow: Account<'info, Escrow>,
    #[account(mut)]
    pub maker: UncheckedAccount<'info>,
    pub system_program: Program<'info, System>,
}
```

**Quasar:**
```rust
#[derive(Accounts)]
pub struct Take<'info> {
    pub taker: &'info mut Signer,
    #[account(
        has_one = maker,
        constraint = escrow.receive > 0,
        close = taker,
        seeds = [b"escrow", maker],
        bump = escrow.bump
    )]
    pub escrow: &'info mut Account<Escrow>,
    pub maker: &'info mut UncheckedAccount,
    pub system_program: &'info Program<System>,
}
```

The constraint syntax is nearly identical. The structural differences are:

| Anchor | Quasar | Why |
|---|---|---|
| `Signer<'info>` | `&'info Signer` or `&'info mut Signer` | References, not owned values. Mutability is expressed in the reference type. |
| `Account<'info, T>` | `&'info Account<T>` or `&'info mut Account<T>` | Same pattern -- references instead of owned wrappers. |
| `#[account(mut)]` attribute | `&'info mut` in the type | Mutability is part of the Rust type, not a separate attribute. The derive macro reads the reference mutability directly. |
| `seeds = [b"escrow", maker.key().as_ref()]` | `seeds = [b"escrow", maker]` | In Quasar, you pass the account field name directly. The macro extracts the address automatically. |

### Business logic placement

In Anchor, instruction logic typically lives in the `#[program]` function body. In Quasar, the convention is to implement methods on the accounts struct:

```rust
impl<'info> Take<'info> {
    #[inline(always)]
    pub fn transfer_tokens(&mut self) -> Result<(), ProgramError> {
        self.token_program
            .transfer(self.taker_ta_b, self.maker_ta_b, self.taker, self.escrow.receive)
            .invoke()
    }
}
```

The `#[program]` block becomes a thin dispatch layer:

```rust
#[instruction(discriminator = 1)]
pub fn take(ctx: Ctx<Take>) -> Result<(), ProgramError> {
    ctx.accounts.transfer_tokens()?;
    ctx.accounts.withdraw_tokens_and_close(&ctx.bumps)
}
```

This keeps each instruction's logic self-contained and testable.

## CPI: Cross-program invocation

**Anchor:**
```rust
let cpi_ctx = CpiContext::new(
    ctx.accounts.token_program.to_account_info(),
    Transfer {
        from: ctx.accounts.vault.to_account_info(),
        to: ctx.accounts.user_token.to_account_info(),
        authority: ctx.accounts.escrow.to_account_info(),
    },
);
token::transfer(cpi_ctx, amount)?;
```

**Quasar:**
```rust
self.token_program
    .transfer(self.vault_ta_a, self.taker_ta_a, self.escrow, amount)
    .invoke_signed(&seeds)?;
```

Quasar's CPI system uses `CpiCall<ACCTS, DATA>`, a const-generic struct where the number of accounts and data size are known at compile time. Everything stays on the stack -- no heap allocation, no `Vec`, no `AccountInfo` cloning.

For system program CPIs like transfers, the pattern is the same:

```rust
self.system_program
    .transfer(self.user, self.vault, amount)
    .invoke()
```

Signed invocation (PDA seeds) uses `.invoke_signed(&seeds)` instead of wrapping in `CpiContext::new_with_signer`.

### PDA seeds for CPI

In Anchor, you construct seed slices manually. In Quasar, the `#[derive(Accounts)]` macro generates a `Bumps` struct with a `*_seeds()` helper method for each PDA:

```rust
pub fn withdraw(&mut self, bumps: &TakeBumps) -> Result<(), ProgramError> {
    let seeds = bumps.escrow_seeds();
    self.token_program
        .transfer(self.vault, self.taker_ta, self.escrow, amount)
        .invoke_signed(&seeds)
}
```

## Events

**Anchor:**
```rust
#[event]
pub struct MakeEvent {
    pub escrow: Pubkey,
    pub maker: Pubkey,
    pub deposit: u64,
}

// In instruction handler:
emit!(MakeEvent {
    escrow: ctx.accounts.escrow.key(),
    maker: ctx.accounts.maker.key(),
    deposit: amount,
});
```

**Quasar:**
```rust
#[event(discriminator = 0)]
pub struct MakeEvent {
    pub escrow: Address,
    pub maker: Address,
    pub deposit: u64,
}

// In instruction handler:
emit!(MakeEvent {
    escrow: *self.escrow.address(),
    maker: *self.maker.address(),
    deposit: amount,
});
```

Quasar events require an explicit discriminator, just like accounts and instructions. The `emit!()` macro emits via `sol_log_data` at roughly 100 CU.

For spoofing-resistant events, Quasar also supports `emit_cpi!()` which emits via a self-CPI to a `__event_authority` PDA. This costs approximately 1,000 CU but proves the event was emitted by your program (the program ID appears in the transaction trace). Supported field types for events are `Address`, `u8`--`u128`, `i8`--`i128`, and `bool`.

## Error codes

**Anchor:**
```rust
#[error_code]
pub enum MyError {
    #[msg("Insufficient funds")]
    InsufficientFunds,
}

// Usage:
require!(amount > 0, MyError::InsufficientFunds);
```

**Quasar:**
```rust
#[error_code]
pub enum MyError {
    #[msg("Insufficient funds")]
    InsufficientFunds,
}

// Usage:
require!(amount > 0, MyError::InsufficientFunds);
```

This is identical. Error variants start at 6000 by default, matching the Anchor convention. You can also specify explicit starting values:

```rust
#[error_code]
pub enum MyError {
    Hello = 0,
    World,           // = 1
    ExplicitNum = 100,
    AfterExplicit,   // = 101
}
```

Quasar also defines framework-level errors in `QuasarError` (starting at 3000) for common validation failures like `AccountNotInitialized`, `InvalidPda`, `ConstraintViolation`, and `HasOneMismatch`.

## `Pubkey` vs `Address`

This is the single most pervasive change when migrating. Anchor uses `Pubkey` (alignment 4, 32 bytes). Quasar uses `Address` (alignment 1, 32 bytes).

Why: Solana account data is not guaranteed to be aligned. Anchor works around this by copying account data into aligned buffers (deserialization). Quasar accesses account data in place (zero-copy), so every field must be alignment-1 to avoid undefined behavior from unaligned reads.

In practice, you replace `Pubkey` with `Address` everywhere:

| Anchor | Quasar |
|---|---|
| `account.key()` | `*account.address()` |
| `Pubkey::new_unique()` | `Address::new_unique()` (tests only) |
| `system_program::ID` | `SYSTEM_PROGRAM_ID` |

The `Address` type supports the same base58 encoding/decoding and comparison operations you are used to. Quasar provides `keys_eq()` for optimized 32-byte comparisons using unaligned u64 word reads.

## `no_std` by default

Anchor programs implicitly use `std`. Quasar programs are `#![no_std]` on-chain (with `std` available during tests via `#![cfg_attr(not(test), no_std)]`).

This means:

- No `String`, `Vec`, or `HashMap` from `std`. Use Quasar's Pod types and fixed-size arrays for on-chain data. Quasar provides its own `String<P, N>` and `Vec<T, P, N>` for dynamic account fields.
- No `println!()` on-chain. Use `solana_program_log::log()` (re-exported as `log` in the prelude) for on-chain logging.
- No `format!()`. For debug builds, enable the `debug` feature to get verbose error messages.

The payoff is that the compiled program contains no allocator, no panic formatting machinery, and no unused standard library code.

## Pod types

For numeric fields, Quasar provides alignment-1 Pod types that can be used in `#[repr(C)]` zero-copy structs:

| Standard type | Quasar Pod type |
|---|---|
| `u16` | `PodU16` |
| `u32` | `PodU32` |
| `u64` | `PodU64` |
| `u128` | `PodU128` |
| `i16` | `PodI16` |
| `i32` | `PodI32` |
| `i64` | `PodI64` |
| `i128` | `PodI128` |
| `bool` | `PodBool` |

In `#[account]` structs, you use standard Rust types (`u64`, `bool`, etc.) and the macro handles the conversion to Pod types in the generated zero-copy companion. You only need Pod types when working with the zero-copy layer directly.

## Migration checklist

Here is a step-by-step process for converting an Anchor program:

1. **Create a new Quasar project** with `quasar init` and copy your `instructions/`, `state.rs`, and `events.rs` files over.
2. **Replace imports:** `anchor_lang::prelude::*` becomes `quasar_lang::prelude::*`.
3. **Replace `Pubkey` with `Address`** everywhere. Replace `.key()` with `*account.address()`.
4. **Add explicit discriminators** to all `#[account]`, `#[instruction]`, and `#[event]` attributes.
5. **Convert account field types** from owned values to references: `Signer<'info>` becomes `&'info Signer`, `Account<'info, T>` becomes `&'info Account<T>`.
6. **Remove `#[account(mut)]` attributes** and use `&'info mut` references instead.
7. **Update PDA seeds** to pass field names directly instead of `.key().as_ref()`.
8. **Convert CPI calls** from `CpiContext` to method-style calls (`.transfer()`, `.mint_to()`, etc.) followed by `.invoke()` or `.invoke_signed()`.
9. **Update events** to use explicit discriminators and `Address` fields.
10. **Update error types** -- these should work as-is if you were already using `#[error_code]`.
11. **Add `#![cfg_attr(not(test), no_std)]`** to `lib.rs` and remove any `std` dependencies from on-chain code.
12. **Build and fix** with `quasar build`. The compiler will catch remaining type mismatches.
