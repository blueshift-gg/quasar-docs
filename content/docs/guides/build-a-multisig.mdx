title: Study the v0.1.0 SOL multisig
description: Trace the example's dynamic state, threshold checks, PDA-signed transfers, and known signer-uniqueness gap.
---

This example stores up to ten signer entries and a threshold in one compact
configuration account. Anyone can deposit SOL. A withdrawal succeeds only when
enough configured signers appear as transaction signers in the instruction's
remaining accounts.

<Callout type="warn" title="Educational example: signer uniqueness is not enforced">
`Remaining<Signer, 10>` permits repeated signer metas. The v0.1.0 `create`
handler copies them without a uniqueness check. If the stored list contains the
same address more than once, one transaction signer can match more than one
stored slot and weaken the intended threshold. The existing duplicate test
covers repeated transaction metas against a unique stored list; it does not
cover duplicate configured signers. Do not use this example to control funds.
</Callout>

<Callout type="warn" title="A transaction multisig, not a proposal system">
The program does not store proposals, approvals, nonces, or timelocks. Required
signers co-sign each transfer transaction directly, and the same authorized set
may approve another transfer later. Add durable proposal state before using
this pattern for delayed or asynchronous governance.
</Callout>

## Keep the signer set bounded

The configuration combines fixed identity fields with bounded dynamic fields:

```rust
#[account(discriminator = 1, set_inner)]
#[seeds(b"multisig", creator: Address)]
pub struct MultisigConfig {
    pub creator: Address,
    pub threshold: u8,
    pub bump: u8,
    pub label: String<32>,
    pub signers: Vec<Address, 10>,
}
```

The PDA seeds allow one configuration per creator. `String<32>` and
`Vec<Address, 10>` keep the account compact while giving validation a fixed
upper bound. Quasar checks their prefixes and payload ranges before exposing
borrowed `label()` and `signers()` views.

```text
[discriminator: 1]
[creator: 32][threshold: 1][bump: 1]
[label length: 1][signer count: 2]
[label bytes: 0..32][signer addresses: 0..320]
```

## Create: validate the list before storing it

`create` uses `CtxWithRemaining<Create>` because signer count is not fixed in
the declared context:

```rust
#[instruction(discriminator = 0)]
pub fn create(
    ctx: CtxWithRemaining<Create>,
    threshold: u8,
) -> Result<(), ProgramError> {
    let signers = ctx.remaining_accounts().parse::<Signer, 10>()?;
    ctx.accounts
        .create_multisig(threshold, &ctx.bumps, signers)
}
```

Typed parsing requires each remaining entry to be a signer and enforces the
ten-account bound. Because `Signer` explicitly permits repeated metas, it does
not establish signer-set uniqueness. The handler copies the validated entries
into a stack buffer and checks the numeric threshold:

```rust
let threshold_count = threshold as usize;
if threshold_count == 0 || threshold_count > count {
    return Err(ProgramError::InvalidArgument);
}

let signers = unsafe { core::slice::from_raw_parts(addrs_ptr, count) };

self.config.set_inner(
    MultisigConfigInner {
        creator: *self.creator.address(),
        threshold,
        bump: bumps.config,
        label: "",
        signers,
    },
    self.creator.to_account_view(),
    self.rent.lamports_per_byte(),
    self.rent.exemption_threshold_raw(),
)
```

The full source uses `MaybeUninit<[Address; 10]>`; its memory-safety argument is
the same bound established by `Remaining<Signer, 10>`. That argument says
nothing about address uniqueness. `set_inner` writes the compact fields and
resizes the account to the exact initial payload.

There is no instruction to change the signer set or threshold after creation.
The creator controls only the label.

## Deposit: separate funding from authorization

The SOL vault is a second PDA derived from the configuration address:

```rust
#[derive(Seeds)]
#[seeds(b"vault", config: Address)]
pub struct MultisigVaultPda;

#[derive(Accounts)]
pub struct Deposit {
    #[account(mut)]
    pub depositor: Signer,
    pub config: Account<MultisigConfig>,
    #[account(mut, address = MultisigVaultPda::seeds(config.address()))]
    pub vault: UncheckedAccount,
    pub system_program: Program<SystemProgram>,
}
```

The depositor does not have to be a configured signer. The System Program moves
their lamports into the system-owned vault; the config account is present only
to bind the vault address to a validated multisig.

## Execute: count configured signers once

Execution parses a fresh bounded signer list, then iterates the stored set
first:

```rust
let stored_signers = self.config.signers();
let threshold = self.config.threshold as u32;

let mut approvals = 0u32;
for stored in stored_signers {
    for signer in signers.iter() {
        if quasar_lang::keys_eq(signer.address(), stored) {
            approvals = approvals.wrapping_add(1);
            break;
        }
    }
    if approvals >= threshold {
        break;
    }
}

if approvals < threshold {
    return Err(ProgramError::MissingRequiredSignature);
}
```

Iterating stored addresses first means repeating one transaction meta does not
increase approvals for one stored slot. It does not protect a configuration
that already contains the same address in multiple stored slots. After the
threshold check, the program signs for the vault PDA and asks the System Program
to transfer to the transaction's chosen recipient:

```rust
let vault_signer = self.vault_signer(bumps);
self.system_program
    .transfer(&self.vault, &self.recipient, amount)
    .invoke_signed(&vault_signer)?;
```

The recipient is not stored in a proposal because this example has no proposal
state. It is authorized by the same transaction signatures that satisfy the
threshold.

## Mutate one dynamic field

`set_label` requires the original creator and binds the same configuration PDA:

```rust
let mut guard = self.config.as_mut(self.creator.to_account_view());
if !guard.label.set(label) {
    return Err(ProgramError::InvalidInstructionData);
}
```

The mutation guard saves on drop. Growing the label takes the rent delta from
the creator; shrinking it returns the excess. The 32-byte bound is checked
before the new payload is committed.

## What the test suite establishes

The pinned QuasarSVM suite covers configuration creation and layout, deposits,
label mutation, a successful threshold transfer, insufficient approvals, and a
repeated transaction signer meta that must not count twice against one stored
slot. It does not test duplicate addresses in the stored configuration, and it
does not contain the invalid-UTF-8 test claimed by the previous page.

| Invariant | Enforced by |
|---|---|
| Signer list is non-empty and at most ten entries | typed remaining parsing plus threshold check |
| Threshold is numerically reachable | `1 <= threshold <= signer_entry_count` |
| Config has one canonical address | `MultisigConfig::seeds(creator.address())` |
| Vault belongs to that config | `MultisigVaultPda::seeds(config.address())` |
| Each approval is a transaction signature | `Remaining<Signer, 10>` |
| Vault transfer needs this program | generated PDA signer plus System Program CPI |

The suite proves those cases for the example. It does not prove configured
signer uniqueness and does not provide proposal replay protection, signer
rotation, spending limits, recipient allowlists, or recovery if the creator
disappears.
