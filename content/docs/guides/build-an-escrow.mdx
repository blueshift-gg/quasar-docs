---
title: Build and verify a token escrow
description: Trace the v0.1.0 escrow's authority, token flow, PDA signing, and terminal states.
---

The escrow exchanges token A from a maker for token B from a taker:

1. `make` records the terms and moves token A into a vault controlled by the
   escrow PDA;
2. `take` pays token B to the stored maker destination, sends all vaulted token
   A to the taker, then closes the vault and escrow; or
3. `refund` lets the maker recover token A and closes both accounts.

The useful part of this example is not the number of accounts. It is how stored
state, token-account validation, and PDA signatures prevent either side from
redirecting the other side's assets.

<Callout type="warn" title="One active escrow per maker">
The escrow address uses only `b"escrow"` and the maker address. A maker cannot
open two concurrent escrows in this design. Add an offer ID to the typed seed
layout before using this as a multi-order protocol.
</Callout>

## Store the terms that later instructions must obey

The escrow PDA records identities and destinations, not token balances:

```rust
#[account(discriminator = 1, set_inner)]
#[seeds(b"escrow", maker: Address)]
pub struct Escrow {
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub maker_ta_b: Address,
    pub receive: u64,
    pub bump: u8,
}
```

`maker_ta_b` is stored because the taker must not choose where the maker's
payment goes. `mint_a`, `mint_b`, and `receive` are the terms later token CPIs
must satisfy. The bump supports the stored-bump verification path after
initialization.

The events use discriminators `1`, `2`, and `3`, separate from instruction and
account discriminator namespaces:

```rust
#[event(discriminator = 1)]
pub struct MakeEvent {
    pub escrow: Address,
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub deposit: u64,
    pub receive: u64,
}

#[event(discriminator = 2)]
pub struct TakeEvent {
    pub escrow: Address,
}

#[event(discriminator = 3)]
pub struct RefundEvent {
    pub escrow: Address,
}
```

## Make: create state before moving assets

The `Make` context creates three things when needed: the escrow state, the
maker's token-B destination, and the token-A vault.

```rust
#[derive(Accounts)]
pub struct Make {
    #[account(mut)]
    pub maker: Signer,
    #[account(init, payer = maker, address = Escrow::seeds(maker.address()))]
    pub escrow: Account<Escrow>,
    pub mint_a: Account<Mint>,
    pub mint_b: Account<Mint>,
    #[account(mut)]
    pub maker_ta_a: Account<Token>,
    #[account(
        init(idempotent),
        payer = maker,
        token(mint = mint_b, authority = maker, token_program = token_program)
    )]
    pub maker_ta_b: Account<Token>,
    #[account(
        init(idempotent),
        payer = maker,
        token(mint = mint_a, authority = escrow, token_program = token_program)
    )]
    pub vault_ta_a: Account<Token>,
    pub rent: Sysvar<Rent>,
    pub token_program: Program<TokenProgram>,
    pub system_program: Program<SystemProgram>,
}
```

The vault's token authority is `escrow`, not the maker. After the deposit, the
maker cannot transfer token A back directly; only a CPI signed by the escrow
PDA can move it.

The handler order is deliberate:

```rust
#[instruction(discriminator = 0)]
pub fn make(ctx: Ctx<Make>, deposit: u64, receive: u64) -> Result<(), ProgramError> {
    ctx.accounts.make_escrow(receive, &ctx.bumps)?;
    ctx.accounts.emit_event(deposit, receive)?;
    ctx.accounts.deposit_tokens(deposit)
}
```

Any failing CPI rolls the transaction back, including the state write and log.
The deposit itself is authorized by the maker's transaction signature:

```rust
self.token_program
    .transfer(&self.maker_ta_a, &self.vault_ta_a, &self.maker, amount)
    .invoke()
```

## Take: bind payment to stored state

The escrow field carries the relationship checks for the terminal trade:

```rust
#[account(
    mut,
    has_one(maker),
    has_one(maker_ta_b),
    constraints(escrow.receive > 0),
    close(dest = taker),
    address = Escrow::seeds(maker.address())
)]
pub escrow: Account<Escrow>,
```

`has_one(maker_ta_b)` prevents the taker from substituting a different token-B
destination. The token behavior on that destination checks mint B and maker
authority. The taker's transfer is then a normal signed CPI for exactly the
stored `receive` amount.

To release token A, the program reconstructs the escrow PDA signer and uses it
for both the transfer and vault closure:

```rust
let bump = [bumps.escrow];
let seeds = [
    Seed::from(b"escrow" as &[u8]),
    Seed::from(self.maker.address().as_ref()),
    Seed::from(bump.as_ref()),
];

self.token_program
    .transfer(
        &self.vault_ta_a,
        &self.taker_ta_a,
        &self.escrow,
        self.vault_ta_a.amount(),
    )
    .invoke_signed(&seeds)?;

self.token_program
    .close_account(&self.vault_ta_a, &self.taker, &self.escrow)
    .invoke_signed(&seeds)
```

The taker receives the vault's entire token-A balance. The vault's rent and the
escrow state's rent also go to the taker in this design; that is protocol policy,
not a framework requirement.

## Refund: require the original maker

Refund replaces the taker with `maker: Signer` and applies
`has_one(maker)` to the escrow. It uses the same PDA signer to return all token A
to a maker-owned token account, closes the vault to the maker, and lets the
context epilogue close the escrow to the maker.

There is no timeout or third-party cancellation path. The maker can refund at
any time before a take transaction succeeds; Solana transaction atomicity
decides which terminal instruction wins if both are submitted.

## What the test suite establishes

The pinned suite executes all three happy paths and checks the exact post-state.
It also supplies pre-existing accounts to the idempotent initialization path
and rejects a maker token-B destination with the wrong mint or authority.

Those tests cover the example's encoded relationships. They do not establish a
fair-price policy, token-extension compatibility, front-running resistance, or
economic safety for arbitrary mints.

| Invariant | Enforced by |
|---|---|
| Only the maker defines an offer | `maker: Signer` during `make` |
| The offer has one canonical state address | `Escrow::seeds(maker.address())` |
| Token B reaches the stored maker destination | `has_one(maker_ta_b)` plus token behavior |
| Only this program releases token A | escrow PDA authority plus signed CPI |
| Only the maker can cancel | `maker: Signer` plus `has_one(maker)` |
| Take and refund are terminal | vault closure and escrow epilogue closure |
