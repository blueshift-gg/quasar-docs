---
title: Build a SOL vault
description: Build and test a system-owned vault PDA whose withdrawals require the program's signature.
---

This vault has one address per user. Deposits are ordinary System Program
transfers into the vault PDA. Withdrawals use the same program, but the Quasar
program signs for the PDA before lamports can leave it.

The distinction matters: receiving a transfer does not make an account owned by
your program. Direct lamport subtraction is therefore the wrong withdrawal path
for this design.

## Create the project

```bash
quasar init quasar-vault --yes --no-git --toolchain solana --test-language rust --rust-framework quasar-svm --template minimal
cd quasar-vault
mkdir -p src/instructions
```

The generated project already contains the QuasarSVM and Solana development
dependencies used by the test below.

The finished source tree is small:

```text
src/
  lib.rs
  tests.rs
  instructions/
    mod.rs
    deposit.rs
    withdraw.rs
```

## Declare the two instructions

Replace `src/lib.rs` with the following program. Keep the `declare_id!` value
generated for your project; the address below belongs to this example.

```rust
#![no_std]

use quasar_lang::prelude::*;

mod instructions;
use instructions::*;
#[cfg(test)]
mod tests;

declare_id!("33333333333333333333333333333333333333333333");

#[program]
mod quasar_vault {
    use super::*;

    #[instruction(discriminator = 0)]
    pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.deposit(amount)
    }

    #[instruction(discriminator = 1)]
    pub fn withdraw(ctx: Ctx<Withdraw>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.withdraw(amount, &ctx.bumps)
    }
}
```

The `#[program]` module only routes decoded instructions. Account policy and
state transitions live on the corresponding accounts structs.

Create `src/instructions/mod.rs`:

```rust
pub mod deposit;
pub use deposit::*;

pub mod withdraw;
pub use withdraw::*;
```

## Deposit into a typed PDA

Create `src/instructions/deposit.rs`:

```rust
use quasar_lang::prelude::*;

#[derive(Seeds)]
#[seeds(b"vault", user: Address)]
pub struct VaultPda;

#[derive(Accounts)]
pub struct Deposit {
    #[account(mut)]
    pub user: Signer,
    #[account(mut, address = VaultPda::seeds(user.address()))]
    pub vault: UncheckedAccount,
    pub system_program: Program<SystemProgram>,
}

impl Deposit {
    #[inline(always)]
    pub fn deposit(&self, amount: u64) -> Result<(), ProgramError> {
        self.system_program
            .transfer(&self.user, &self.vault, amount)
            .invoke()
    }
}
```

The context establishes the authority boundary before the CPI:

- `user` signed and is writable because its balance decreases;
- `vault` is writable and must equal the PDA derived from `"vault"` and that
  signer; and
- `system_program` is the executable System Program.

`UncheckedAccount` is intentional here. The vault has no program data to
decode; its address and writability are the relevant checks.

## Withdraw with the PDA signature

Create `src/instructions/withdraw.rs`:

```rust
use {super::deposit::VaultPda, quasar_lang::prelude::*};

#[derive(Accounts)]
pub struct Withdraw {
    #[account(mut)]
    pub user: Signer,
    #[account(mut, address = VaultPda::seeds(user.address()))]
    pub vault: UncheckedAccount,
    pub system_program: Program<SystemProgram>,
}

impl Withdraw {
    #[inline(always)]
    pub fn withdraw(
        &self,
        amount: u64,
        bumps: &WithdrawBumps,
    ) -> Result<(), ProgramError> {
        let vault_signer = self.vault_signer(bumps);
        self.system_program
            .transfer(&self.vault, &self.user, amount)
            .invoke_signed(&vault_signer)
    }
}
```

The System Program owns the vault, so it performs the outgoing transfer and
enforces the balance check. The transfer also requires the source account to
sign. `vault_signer` reconstructs the seed set already validated for the
`vault` field, allowing the runtime to sign for that PDA and no other address.

Requiring `user: Signer` means a third party cannot trigger a withdrawal even
though the destination is fixed to the user whose key is in the PDA seeds.

## Test the complete lifecycle

Replace `src/tests.rs` with one test that feeds the deposited account state into
the withdrawal:

```rust
extern crate std;

use {
    quasar_svm::{Account, Pubkey, QuasarSvm},
    solana_address::Address,
    solana_instruction::{AccountMeta, Instruction},
    std::vec,
};

fn setup() -> QuasarSvm {
    let elf = std::fs::read("target/deploy/quasar_vault.so").unwrap();
    QuasarSvm::new().with_program(&Pubkey::from(crate::ID), &elf)
}

fn system_account(address: Pubkey, lamports: u64) -> Account {
    Account {
        address,
        lamports,
        data: vec![],
        owner: quasar_svm::system_program::ID,
        executable: false,
    }
}

fn vault_instruction(
    discriminator: u8,
    user: Pubkey,
    vault: Pubkey,
    amount: u64,
) -> Instruction {
    let mut data = vec![discriminator];
    data.extend_from_slice(&amount.to_le_bytes());

    Instruction {
        program_id: Address::from(crate::ID.to_bytes()),
        accounts: vec![
            AccountMeta::new(Address::from(user.to_bytes()), true),
            AccountMeta::new(Address::from(vault.to_bytes()), false),
            AccountMeta::new_readonly(
                Address::from(quasar_svm::system_program::ID.to_bytes()),
                false,
            ),
        ],
        data,
    }
}

#[test]
fn deposit_then_withdraw() {
    let mut svm = setup();
    let user = Pubkey::new_unique();
    let (vault, _) =
        Pubkey::find_program_address(&[b"vault", user.as_ref()], &crate::ID);

    let deposit = vault_instruction(0, user, vault, 1_000_000_000);
    let deposited = svm.process_instruction(
        &deposit,
        &[system_account(user, 10_000_000_000), system_account(vault, 0)],
    );
    deposited.assert_success();
    assert_eq!(deposited.account(&vault).unwrap().lamports, 1_000_000_000);

    let withdraw = vault_instruction(1, user, vault, 400_000_000);
    let withdrawn = svm.process_instruction(&withdraw, &deposited.accounts);
    withdrawn.assert_success();

    assert_eq!(withdrawn.account(&user).unwrap().lamports, 9_400_000_000);
    assert_eq!(withdrawn.account(&vault).unwrap().lamports, 600_000_000);
}
```

Build the SBF binary and run the test through the project configuration:

```bash
quasar test
```

The test is deliberately sequential. Two independent tests, one with a
system-owned deposit vault and one with a pre-funded program-owned withdrawal
vault, would not prove that the documented lifecycle works.

## What this design guarantees

- Every user gets one deterministic vault address under this program ID.
- Only a transaction signed by that user can request a withdrawal.
- Only this program can produce the PDA signature required for an outgoing
  System Program transfer.
- The System Program rejects a withdrawal larger than the vault balance.

This vault has no persistent data, delegate, lockup, or recovery authority. Use
an `Account<T>` state PDA when the withdrawal policy needs more than the user's
signature.
